### 2.1 2025-2026年RAG新范式

#### 2.1.1 GraphRAG (微软)

**核心理念**: 将知识图谱引入RAG

```python
# 典型GraphRAG流程
1. 文档 -> 实体提取 -> 知识图谱
2. 社区检测 -> 生成社区摘要
3. 查询 -> 社区匹配 -> 答案生成
```

**优势**:
- 回答"全局性问题"(如"概述...")
- 保留实体关系上下文
- 减少幻觉

**最新版本**: GraphRAG 2.0 (2025)
- 动态图谱更新
- 支持多模态
- 更好的社区检测算法

**适用性评估**: 7/10
- 适合需要关系推理的场景
- 开源实现: microsoft/graphrag

#### 2.1.2 HyDE (Hypothetical Document Embeddings)

**原理**:
1. 用LLM生成"假设性回答"
2. 用假设性回答的embedding去检索
3. 结合原始query检索结果

```python
# HyDE流程示例
query = "如何学习Python"
hypothetical_doc = llm.generate("用3句话回答如何学习Python")
query_embedding = embed(hypothetical_doc)
results = vector_db.search(query_embedding)
```

**效果提升**:
- Recall提升15-30%
- 对模糊查询效果显著

**适用性评估**: 8/10
- 简单易实现
- 效果稳定

---

#### 2.1.3 BM25+向量混合检索

**原理**: 结合关键词精确匹配 + 语义相似度

| 检索方式 | 优势 | 劣势 |
|---------|------|------|
| BM25 | 精确关键词匹配 | 无法语义理解 |
| 向量检索 | 语义相似度 | 对专有名词不敏感 |
| **混合** | 两者兼顾 | 实现复杂度高 |

**最新实践**:
- Reciprocal Rank Fusion (RRF)
- `score = 0.7 * vector_sim + 0.3 * bm25_score`
- 权重可调

**适用性评估**: 9/10 (强烈推荐)

#### 4.1.2 Second Brain架构

**三层记忆模型**:

| 层级 | 容量 | 访问速度 | 内容类型 |
|------|------|---------|---------|
| Working Memory | ~128K | 即时 | 当前任务相关 |
| Episodic Memory | ~1M | 100ms | 最近交互 |
| Semantic Memory | 无限 | 1-5s | 长期知识 |

**实现要点**:
```python
class SecondBrain:
    def __init__(self):
        self.working = WorkingMemory()  # LLM上下文
        self.episodic = EpisodicMemory()  # 向量存储
        self.semantic = SemanticMemory()   # 知识图谱

    def recall(self, query):
        # 层级检索
        recent = self.working.search(query)
        if not recent:
            recent = self.episodic.search(query)
        if not recent:
            recent = self.semantic.search(query)
        return recent
```

**适用性评估**: 8/10

### 5.2 RAG vs 长上下文窗口的取舍

#### 详细对比

| 维度 | RAG | 长上下文 |
|------|-----|---------|
| **成本** | 低 (按检索量) | 高 (按token) |
| **延迟** | 低 (100-500ms) | 高 (10-60s) |
| **准确性** | 依赖检索质量 | 依赖模型能力 |
| **可解释性** | 高 (可追溯来源) | 低 (黑盒) |
| **幻觉风险** | 较低 | 较高 |
| **更新频率** | 高 (增量更新) | 低 (需重推理) |
| **大规模文档** | 支持无限 | 受限 |

#### 决策树

```
查询类型是什么?
├── 简单事实查询 → RAG (BM25+向量)
├── 关系推理查询 → GraphRAG
├── 需要全文理解 → 长上下文模型
└── 两者结合 → 混合方案 (RAG召回 + LLM精调)
```

**强烈推荐**: 混合方案 (RAG召回 + 长上下文精调)

```python
# 混合方案示例
class HybridRAG:
    def retrieve(self, query):
        # Step 1: RAG召回top-k相关chunk
        rag_results = self.rag.retrieve(query, top_k=20)

        # Step 2: 构建上下文
        context = "\n".join([r.content for r in rag_results])

        # Step 3: 如果context过长，压缩
        if len(context) > 128_000:
            context = self.compressor.compress(context)

        # Step 4: 生成回答
        return self.llm.generate(context, query)
```

### 5.3 避免RAG检索"丢失"问题

#### 丢失原因分析

| 原因 | 表现 | 解决方案 |
|------|------|---------|
| 切片不当 | 关键信息被切断 | 语义切片+重叠 |
| 向量偏差 | 相似但不相关 | 混合检索 |
| 噪声干扰 | 低相关度内容 | 重排序+压缩 |
| 查询不匹配 | 表达方式不同 | HyDE/Query改写 |

#### 完整解决方案

```python
class RobustRAG:
    def __init__(self):
        self.hybrid_retriever = HybridRetriever(
            sparse_weight=0.3,  # BM25权重
            dense_weight=0.7,  # 向量权重
        )
        self.query_rewriter = QueryRewriter()  # Query改写
        self.reranker = CrossEncoderReranker()  # 重排序
        self.compressor = LLMLinguaCompressor() # 压缩

    def retrieve(self, query, top_k=10):
        # 1. Query改写
        rewritten_query = self.query_rewriter.rewrite(query)

        # 2. 多路检索
        results = self.hybrid_retriever.retrieve(rewritten_query)

        # 3. 重排序
        reranked = self.reranker.rerank(query, results, top_k=5)

        # 4. 上下文压缩
        context = self.compressor.compress(reranked)

        return context
```

#### 关键技巧

1. **增加查询多样性**
   - 同一问题多种表述
   - 合并多次检索结果

2. **使用元数据过滤**
   - 时间、来源、类型
   - 减少噪声

3. **验证检索结果**
   - 让LLM判断相关性
   - 反馈循环优化

4. **保留检索轨迹**
   - 记录每个答案的来源
   - 支持人工验证
