# AI教材编写系统 — 出版规范设计

> 本文档从出版行业规范角度，为AI教材编写系统提供逻辑架构、引用规范、术语管理、审核机制的设计方案。

---

## 一、教材逻辑架构设计

### 1.1 章节依赖关系的形式化表示

采用类似YAML frontmatter的结构化元数据，嵌入每个章节文档头部：

```yaml
---
chapter_id: ch_04_03
title: 无人机空域管理法规
level: 3
order: 4.3

# 前置依赖（必须先完成才能开始本章）
prerequisites:
  chapters:
    - ch_02_01  # 航空法基础
    - ch_03_02  # 低空空域分类
  concepts:
    - "空域分类体系"     # 前文必须明确建立的概念
    - "无人机适航要求"   # 可从同章前节引用

# 本章产出的概念（供后续章节引用）
outputs:
  concepts:
    - "UTM系统"
    - "轻小型无人机运行规范"
  data:
    - "表4-3-1: 轻小型无人机分类标准"
  conclusions:
    - "结论4-3-2: 融合空域管理最优路径"

# 章节内的引用锚点（本章内部引用）
internal_refs:
  - anchor: "图4-3-1"
    target: "ch_04_03#fig_432"  # 同一章内其他节
  - anchor: "式(4-3-1)"
    target: "ch_04_03#eq_431"

# 跨章节引用锚点（被前章引用时记录）
cross_refs_from:
  - ch_05_02  # 第五章会引用本章结论
  - ch_06_01  # 第六章会引用本章数据

# 逻辑流角色
role_in_flow:
  - "建立概念"    # 知识引入型
  - "提供依据"    # 法规引用型
  - "连接前后"    # 过渡型
---
```

**依赖关系检查规则：**
- AI写作启动前，自动检查所有`prerequisites`是否已满足（章节存在 + 概念已定义）
- 若依赖未满足，阻塞本章写作，提示"概念X尚未在ch_XX_XX中建立"
- 循环依赖检测：禁止形成A→B→C→A的闭环

### 1.2 逻辑顺序的判断标准

#### 三维排序矩阵

| 维度 | 权重 | 判断依据 |
|------|------|----------|
| **概念先修** | 40% | 本章引入的概念是否依赖前文的概念定义 |
| **难度递进** | 35% | 本章内容是否在认知负荷上高于前文 |
| **主题聚类** | 25% | 本章是否与某几个前章属于同一主题模块 |

#### 概念先修判断算法

```
function check_prerequisite(concept_A, concept_B):
    # A是否需要先于B建立
    # 条件：定义B时需要引用A的解释/数据/结论

    has_direct_def_ref = B.refs.contains(A.concept_id)
    has_applied_method = B.uses(A.methodology)
    has_data_dependency = B.requires(A.dataset)

    return has_direct_def_ref OR has_applied_method OR has_data_dependency
```

#### 难度递进的量化指标

| 难度层级 | 特征 | 示例 |
|----------|------|------|
| L1 事实性 | 定义、分类、辨认 | "低空空域分为三类" |
| L2 解释性 | 原理阐述、因果分析 | "为何低空空域需要特殊管理" |
| L3 应用性 | 案例分析、规则适用 | "某场景下如何申请飞行许可" |
| L4 评估性 | 比较分析、方案权衡 | "两种空域管理模式的优劣对比" |
| L5 创造性 | 模式设计、策略创新 | "设计一套适应中国国情的UTM系统" |

**升层规则：** 同一主题模块内，章节难度应单调递增或平行（不允许L4突然降到L2）

#### 主题聚类识别

```yaml
modules:
  - id: mod_airspace
    title: 空域管理
    chapters: [ch_02_01, ch_03_02, ch_04_03, ch_05_01]
    coherence_rule: "线性递进——从法理→分类→管理→技术"

  - id: mod_technology
    title: 无人机技术
    chapters: [ch_03_01, ch_04_02, ch_06_02]
    coherence_rule: "横向并列——技术维度并行展开"
```

**模块间顺序规则：**
- 跨模块引用必须通过"桥梁章节"（专门用于承接前一模块结论的章节）
- 禁止模块间隐式依赖（不能假设读者记住另一模块的内容）

### 1.3 全局逻辑链的核心组成要素

```
┌─────────────────────────────────────────────────────────────┐
│                    全局逻辑链 (GLC)                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 核心问题链                                               │
│     Q0 (总问题) → Q1 → Q2 → ... → Qn                       │
│     "低空经济如何实现安全有序发展？"                           │
│      ↓                                                      │
│     "需要解决哪些核心矛盾？"                                  │
│      ↓                                                      │
│     "每个矛盾的解决路径是什么？"                              │
│                                                             │
│  2. 概念演进链                                              │
│     C0 (基础概念) → C1 → C2 → ... → Cn (综合概念)           │
│     "低空" → "低空空域" → "空域分类" → "融合空域"            │
│                                                             │
│  3. 法规依据链                                               │
│     上位法 → 行政法规 → 部门规章 → 地方规范                  │
│     《民航法》 → 《无人驾驶航空器飞行管理条例》 → CCAR-92    │
│                                                             │
│  4. 方法论链                                                │
│     基础理论 → 分析方法 → 应用框架 → 实践工具                │
│                                                             │
│  5. 案例贯穿线                                              │
│     [案例档案] 贯穿全书的真实案例，                          │
│     每个模块都用同一案例的不同切面                              │
│     "深圳无人机管理办法" 从第2章法理分析                      │
│      → 第4章实践应用 → 第6章问题诊断 → 第8章改进建议          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**逻辑链的显式化要求：**

每个章节必须在其元数据中声明：

```yaml
global_chain_contribution:
  problem_chain:
    - question: "Q2: 低空空域管理权责如何划分"
      answered_in: "本章节 ch_04_03"
  concept_chain:
    - concept: "融合空域"
      evolves_from: "ch_03_02的'分类空域'"
      next_step: "ch_05_01的'动态空域'"
  case_perspective:
    - case_id: "case_sz_drone"
      this_chapter_focus: "深圳案例的空域申请流程"
```

## 二、引用规范设计

### 2.1 显式引用与隐式引用划分

#### 必须显式引用的内容

| 类别 | 判断标准 | 引用格式示例 |
|------|----------|--------------|
| **数据/统计** | 具体数值 | "根据民航局2024年数据，全国注册无人机达XX架（来源：民航局官网，2024-06）" |
| **法规条文** | 法律/法规/标准原文 | "依据《无人驾驶航空器飞行管理条例》第十七条" |
| **权威结论** | 学术权威/监管部门明确的结论 | "中国民航局明确指出，UTM系统应采用..." |
| **特定案例** | 具体的、不可泛化的案例 | "深圳南山区2023年'9·2'无人机伤人事件" |
| **方法论引用** | 引用他人提出的分析框架 | "采用Smith等（2022）提出的三维度评估模型" |

#### 可以隐含引用的内容

| 类别 | 判断标准 | 处理方式 |
|------|----------|----------|
| **通用常识** | 跨领域读者普遍了解 | "无人机通过无线电通信与地面站交互" |
| **前文已建立的概念** | 在本书中已正式定义 | 引用概念名称，不引用具体位置 |
| **行业惯例** | 无争议的常规做法 | 直接陈述，不标注来源 |
| **基础逻辑** | 演绎推理的中间步骤 | 不引用，直接使用 |

**隐含引用的"保质期"规则：**

```
隐含引用有效性 = 前文概念的"确定性等级"

确定性等级：
  A+ 级：已在专章中详细定义的概念 → 全书有效
  A  级：在某节中定义的概念 → 在定义该章+后续1章内有效
  B  级：仅在某个知识点中提及 → 仅在同页/同段内有效
  C  级：旁白/举例中提到的 → 仅在当页内有效
```

### 2.2 跨章节引用的时效性管理

```yaml
cross_chapter_citation:
  # 引用方向：后→前（本章引用前文）
  backward:
    max_age: 3  # 最多引用前3章的内容
    exception:
      - type: "核心概念"
        chapters_back: 99  # 允许引用全书任何位置的 core_concepts
      - type: "法规条文"
        chapters_back: 99  # 允许引用法律依据章节

  # 引用方向：前→后（为后续章节埋下伏笔）
  forward:
    allowed: true
    constraint: "必须标注'详见第X章'，不允许隐式假设"
    forward_ref_example: "关于UTM系统的技术细节，详见第6章第2节"
```

**引用保质期形式化检查：**

```python
def check_citation_validity(citation, current_chapter, all_chapters):
    ref_target = citation.target  # 被引用的章节

    if citation.type == "core_concept":
        return True  # 核心概念无时效限制

    distance = current_chapter.order - ref_target.order

    if citation.type == "law_reference":
        return distance >= 0  # 法规只能引用早于等于本章的

    if distance > 3:
        return {
            "valid": False,
            "reason": f"引用距离{distance}章，超过3章上限",
            "suggestion": "在本章重新简要提示被引概念，或将引用关系改为'详见第X章'"
        }

    return {"valid": True}
```

### 2.3 引用格式规范

#### 章节内引用格式

| 引用类型 | 格式 | 示例 |
|----------|------|------|
| 图表引用 | `图X-Y` / `表X-Y` / `式(X-Y)` | "如图3-2-1所示" |
| 结论引用 | `结论X-Y-Z` | "基于结论4-2-3" |
| 概念定义引用 | `参见第X章第Y节` | "UTM系统（参见第6章第2节）" |
| 案例引用 | `案例档案X` | "深圳案例（案例档案A-1）" |

#### 外部引用格式（脚注）

```
引用类型        格式
─────────────────────────────────────────
法规            [法规名称，施行日期]
学术文献        [作者，年份] 置于句末
数据来源        [来源机构，发布日期]
网络资源        [网站名称，访问日期]（仅用于补充阅读）
```

## 三、术语一致性保障

### 3.1 术语表的必要字段

```yaml
# 术语表条目结构
term_entry:
  term_id: TERM_0042

  # 术语本体
  canonical_form: "统一融合空域管理"
  pinyin: "tongyi yuhuan kongyu guanli"
  english: "Unified Traffic Management (UTM)"

  # 定义（锁定后不可修改）
  definition:
    content: "指在融合空域内，由军方、民航、地方多头管理向统一平台协同管理的模式转变"
    locked: true
    locked_at: "ch_04_03"  # 首次定义的章节
    locked_version: 1

  # 出现位置追溯
  occurrences:
    - chapter: "ch_04_03"
      context: "首次正式定义"
      status: "definition"  # 定义出现
    - chapter: "ch_05_01"
      context: "应用讨论"
      status: "use"
    - chapter: "ch_06_02"
      context: "对比分析"
      status: "use"
      note: "与'分布式空域管理'对比"

  # 同义词（废弃列表）
  deprecated_synonyms:
    - "一体化的空域管理"  # 原因：语义不完全等同
    - "统一空域管理"     # 原因：忽略了'融合'的内涵

  # 禁止使用的表述
  forbidden_usages:
    - "UTMS"          # 理由：缩写不标准
    - "统一管理空域"   # 理由：语序不当，语义偏移

  # 相关术语（形成语义网络）
  related_terms:
    - term_id: "TERM_0041"
      relation: "上位概念"
      term: "空域管理"
    - term_id: "TERM_0051"
      relation: "对比概念"
      term: "分布式空域管理"
```

### 3.2 AI写作时的术语检查机制

#### 写作前的"术语就绪"检查

```yaml
pre_write_check:
  enabled: true

  for_chapter: "ch_05_02"

  required_terms:
    - term_id: "TERM_0042"
      status: "ready"  # 已在ch_04_03定义
      can_use: true

    - term_id: "TERM_0055"
      status: "not_ready"
      block_reason: "该术语尚未在任何章节定义"
      suggestion: "在ch_05_02中定义，或确认是否引用了前文未建立的概念"

  optional_terms:
    - term_id: "TERM_0071"
      status: "ready"
      suggestion: "建议使用该术语以保持全书一致性"
```

#### 写作中的实时术语监控

```
AI写作输出 → 术语提取 → 术语比对引擎

检测规则：
  1. 若出现 deprecated_synonyms 中的词 → 警告：建议使用正式术语
  2. 若出现 forbidden_usages 中的词 → 错误：明确禁止使用
  3. 若出现未收录的新词 → 提示：建议添加到术语表
  4. 若同一概念出现多种表述 → 警告：术语不一致
```

#### 术语一致性报告

```yaml
terminology_audit:
  chapter: "ch_05_02"
  timestamp: "2025-01-15T10:30:00Z"

  issues:
    - severity: "error"
      type: "deprecated_usage"
      text: "采用'UTMS系统'的表述"
      suggestion: "应使用'UTM系统'"
      position: "第3段第2句"

    - severity: "warning"
      type: "inconsistent_reference"
      text: "本章引用'UTM'时，时而全称时而缩写"
      suggestion: "统一使用'统一融合空域管理(UTM)'"
      positions: ["第2段", "第5段", "第8段"]

  compliance_score: 85
  pass_threshold: 90
  status: "FAILED"
```

### 3.3 术语定义的"锁定"机制

#### 锁定规则

```yaml
term_locking:
  # 定义一旦锁定，不可修改内容，只能扩展
  lock_trigger:
    - event: "章节首次正式定义"
      scope: "该章节内所有首次定义的概念"

    - event: "专家评审通过"
      scope: "评审涉及的章节中的所有定义"

  lock_properties:
    term_id: 不可修改
    canonical_form: 不可修改（若需修改走"术语变更"流程）
    definition.content: 严格锁定，不可修改措辞
    definition.scope: 可在后续章节中补充"在本章语境下的延伸义"

  # 锁定后的变更流程
  change_process:
    name: "术语变更申请"
    steps:
      1. 在变更申请单中说明：
         - 原术语ID和新术语ID
         - 变更原因
         - 影响章节分析（哪些章节的哪些引用需要更新）
      2. 影响评估：
         - 若影响章节 ≤ 3章 → 主编审批
         - 若影响章节 > 3章 → 专家委员会审批
      3. 变更执行：
         - 原术语ID标记为 deprecated
         - 新术语ID正式启用
         - 全局搜索替换（AI辅助）
      4. 变更记录：
         - 记录于术语变更日志
         - 通知相关章节作者
```

#### 锁定状态查询

```python
def get_term_lock_status(term_id):
    term = glossary[term_id]

    if not term.definition.locked:
        return {
            "status": "UNLOCKED",
            "can_define_in": "任何章节",
            "risk": "可能被后续章节覆盖定义"
        }

    locked_chapter = term.definition.locked_at

    return {
        "status": "LOCKED",
        "locked_at": locked_chapter,
        "can_be_extended": True,  # 可补充语境延伸义
        "cannot_be": ["修改定义核心内容", "重新锁定到其他章节"],
        "deprecation": "需走变更流程"
    }
```

## 五、附录：AI辅助工具建议

### 5.1 概念依赖图谱工具

自动生成章节依赖可视化，确保无循环依赖：

```
输入：所有章节的prerequisites元数据
输出：有向无环图(DAG) + 拓扑排序结果
异常检测：循环依赖、孤岛章节（无任何依赖/被依赖）
```

### 5.2 跨章节引用追踪仪表盘

实时监控跨章节引用状态：

```
引用健康度指标：
  - 有效引用数 / 总引用数
  - 超时引用数（距原章节>3章）
  - 前向引用数（对后续章节的引用）

预警规则：
  - 当某章被引用次数 > 10次 → 提示考虑是否为"关键章节"过度集中
  - 当某章的外部引用全部来自相邻2章 → 提示是否可整合
```

### 5.3 术语一致性实时扫描

在AI写作过程中嵌入：

```
扫描频率：每输出200字进行一次
扫描内容：
  1. 新出现的术语 → 与术语表比对
  2. 已定义术语 → 检查是否使用canonical_form
  3. 疑似新术语 → 提示是否添加到术语表

输出：实时术语合规度评分
```
