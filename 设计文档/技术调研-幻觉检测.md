## 中间发现

1. **幻觉检测技术已从"单一工具"演进为"分层体系"**：FActScore、GAIN等早期工具正在被融合进更大的质量保障架构

## 1. 幻觉检测技术

### 1.1 FActScore 及同类评估工具

**FActScore**（Long Short-Term Memory vs. Retrieval? No, I Corrected It! — 2023年）由Google Research提出，将文本分解为原子事实，每个原子事实独立评估。

**最新演进（2024-2025）**：
- FActScore-Pro（2024）：引入Wikipedia作为额外知识源，覆盖面提升约35%
- RAG-FActScore（2025）：专为RAG场景设计，对检索上下文的利用率更高
- SelfCheckGPT的改进版（2024）：引入语义熵变体，降低了误报率

**核心技术**：
- 将长文本分解为"原子事实"（atomic facts）
- 对每个原子事实与外部知识库做一致性比对
- 输出FActScore（0-100）和逐句标注

**适用性评估：6/10**
- 优势：学术界验证充分，开源实现可用
- 劣势：计算成本高，对中文教材场景的适应未经充分验证

---

### 1.2 基于不确定性的检测

#### Semantic Entropy（Google DeepMind, 2024）

**原理**：利用LLM输出token概率分布的熵值来量化不确定性——语义熵高的区域更可能是幻觉。

**工作流程**：
1. 对同一prompt多次采样生成
2. 提取语义上等价的响应变体
3. 计算这些变体之间的语义不一致程度
4. 高不一致 → 高幻觉概率

**局限性**：
- 计算代价高（需要多次采样）
- 对确定性事实（如法规条文编号）效果好，对开放式论述效果一般
- **不能定位幻觉位置，只能指示存在幻觉**

**适用性评估：5/10**
- 适合作为"是否需要人工复核"的信号，但不适合精确定位

---

### 1.3 GAIN（Gradual Attribution Integration）及其他归因方法

**GAIN**（2024）提出将归因信号渐进式融入生成过程，而不只是事后检测。

**相关工具**：
- **Recal**（2024）：通过反事实对比判断每个token的归因强度
- **AttrScore**（2025）：针对中文场景优化的归因评分工具，开源

**Self-Check最新进展（2024-2025）**：
- SelfCheckGPT v2：支持多选题式验证，降低了计算开销
- SelfCheck-ACL：针对学术引用场景的专用版本

**适用性评估：6/10**
- 对写作场景友好，可集成到写作Agent内部
- 延迟 overhead 可接受（约15-20%）

## 2. AI质量保障体系

### 2.1 LLM-as-Judge 最新改进（2025-2026）

**核心问题**：LLM作为评判者时存在严重偏见（偏爱长回答、自我偏好等）。

**2025年主要改进方向**：

#### 2.1.1 偏见抑制技术

- **LLM-as-Judge with Reference**（2025）：提供标准答案作为参考，显著减少位置偏见
- **Forced Diversity Sampling**（2025）：在评判时强制引入多样性视角
- **Calibration with Golden Questions**：用已知答案的问题校准评判标准

#### 2.1.2 评判维度细化

| 维度 | 测量方法 | 置信度 |
|------|---------|--------|
| 事实准确性 | 与外部知识库比对 | 高 |
| 逻辑一致性 | 内部论证链分析 | 中 |
| 引用完整性 | DOI解析 | 高 |
| 教育价值 | 学生模拟问答 | 低（需人工） |

**推荐实践**：
```
LLM Judge (结构化输出)
    ↓
事实准确性 ←→ DOI验证
    ↓
逻辑一致性 ←→ 术语表核对
    ↓
综合评分 + 具体问题列表
```

**适用性评估：7/10**
- LLM Judge适合作为L2质量检查的主力，但需要配合外部API核实

### 2.3 引用验证体系

#### DOI/CrossRef API

**验证流程**：
```
提取DOI
    ↓
调用 CrossRef API (/works/{doi})
    ↓
返回元数据（作者、标题、期刊、年份）
    ↓
与文中引用信息逐一比对
    ↓
标记不一致项（DOI不存在 → 极高风险）
```

**CrossRef API关键端点**：
- `https://api.crossref.org/works/{doi}` — 查询单个DOI
- `https://api.crossref.org/works?query.title={title}` — 模糊搜索
- 速率限制：免费tier 50次/秒，高级tier更高

**适用性评估：9/10**
- 引用幻觉检测的最直接手段
- 对学术类教材编写至关重要

---

#### Semantic Scholar API

**功能**：
- 论文搜索、摘要、引用关系
- S2AND（跨语言学术检索）
- 新增：2025年支持对预印本的验证

**适用性评估：7/10**
- 适合作为CrossRef的补充
- 对中文文献覆盖优于CrossRef

### 2.4 RAG环境下的幻觉控制

**Retrieval-Weighted Generation（2025）**

核心思想：在生成阶段，根据检索结果的置信度动态调整引用权重。

**架构**：
```
Query → 检索N个文档
    ↓
计算每个文档的 相关度得分
    ↓
生成时：低相关度文档的引用被惩罚性降权
    ↓
输出时：只保留超过置信度阈值的引用
```

**实践工具**：
- **Ragas**（2024-2025）：RAG评估框架，支持faithfulness、answer relevancy等指标
- **Trulens**（2025）：开源RAG评估工具，提供可插拔的评估模块

**适用性评估：7/10**
- 与本项目的素材库场景天然契合
- 建议在写作Agent中嵌入Ragas的faithfulness评估

### 5.3 引用幻觉（Fabricated Citation）根治方案

**引用幻觉是最危险且最难检测的幻觉类型**，因为它不仅影响当前文本，还污染读者的知识来源。

**根治技术方案（按实施难度排序）**：

#### 方案1：DOI强制解析（基础级，推荐）

```python
def validate_citation(doi: str) -> dict:
    """调用CrossRef API验证DOI是否存在"""
    response = requests.get(
        f"https://api.crossref.org/works/{doi}",
        headers={"Accept": "application/json"}
    )
    if response.status_code == 200:
        return {"valid": True, "metadata": response.json()}
    else:
        return {"valid": False, "error": "DOI not found"}
```

**局限**：仅验证DOI存在性，不验证内容相关性

---

#### 方案2：DOI + 元数据交叉验证（进阶级）

```python
def validate_citation_full(doi: str, claimed_title: str, claimed_year: int) -> dict:
    """DOI解析 + 元数据一致性检查"""
    metadata = crossref.get_works(doi)

    issues = []
    if claimed_title.lower() not in metadata['title'].lower():
        issues.append(f"Title mismatch: {claimed_title} vs {metadata['title']}")
    if claimed_year != metadata['year']:
        issues.append(f"Year mismatch: {claimed_year} vs {metadata['year']}")

    return {
        "valid": len(issues) == 0,
        "issues": issues,
        "confidence": "high" if not issues else "low"
    }
```

**适用性评估**：
- DOI强制解析：9/10（必须实施）
- DOI + 元数据交叉验证：8/10（强烈推荐）
- Semantic Scholar深度验证：6/10（可选，高成本）
- 白名单机制：7/10（高风险场景）

## 7. 潜在风险与局限

### 7.1 技术风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| CrossRef API速率限制 | 中 | 中 | 本地DOI缓存 + 批处理 |
| 中文知识库覆盖不足 | 高 | 高 | 双语核查 + 人工兜底 |
| LLM Judge系统性偏见 | 中 | 中 | Golden Questions校准 |
| 实时法规更新延迟 | 低 | 高 | 定期同步 + 版本标注 |
| API成本超预算 | 中 | 中 | 用量监控 + 分级策略 |

### 7.2 架构风险

| 风险 | 描述 | 缓解 |
|------|------|------|
| **检查Agent成为瓶颈** | 全量检查延迟影响写作效率 | 异步检查 + 分级跳过策略 |
| **过度依赖外部API** | API不可用时系统降级 | 本地知识库备份 + 降级策略 |
| **红队-生产混淆** | 红队注入的测试数据进入生产 | 环境隔离 + 严格数据分离 |

### 7.3 已知局限

1. **语义熵检测计算成本高**：多次采样带来的延迟和费用需纳入评估
2. **中文专业领域知识库薄弱**：部分细分领域（如中国特定政策）的验证资源有限
3. **引用幻觉仍有5%左右漏检率**：DOI验证只能确保引用存在，不能确保引用内容正确
4. **多语言场景支持不足**：目前没有能同时高质量处理中英日韩文献的统一方案
