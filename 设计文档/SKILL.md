## 1. 项目概述

### 1.1 项目目标

设计一个支持20万字以上教材编写的AI多Agent技能系统，包含调研、专家研讨、提纲生成、分章节写作、排版设计、合规性检查等功能，并解决LLM上下文限制和幻觉问题。

### 1.2 核心约束

| 约束项 | 说明 |
|--------|------|
| 教材规模 | 20万字以上（约100万tokens） |
| 上下文限制 | 单次LLM上下文窗口有限，无法一次性处理全书 |
| 幻觉问题 | LLM在长文本场景下幻觉概率显著增加 |
| 质量要求 | 出版级标准，需通过合规性审查 |
| 人工介入 | 提纲审核、关键章节初审必须人工执行 |

### 1.3 核心挑战

1. **上下文限制**：20万字约100万tokens，无法一次性装入LLM上下文
2. **逻辑一致性**：切片方案会导致章节间逻辑性丢失
3. **幻觉控制**：长文本场景下幻觉概率显著增加
4. **可追溯性**：从源材料到最终内容必须可追溯

## 2. 系统架构

### 2.1 四层Agent架构

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                            L1 编排层（Orchestration Layer）              │
│  工作流引擎 │ 状态机 │ 人工介入调度 │ 错误处理与回滚                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                            L2 Agent层（Agent Layer）                     │
│  调研Agent │ 写作Agent │ 检查Agent │ 合规Agent │ 出版Agent              │
├─────────────────────────────────────────────────────────────────────────────┤
│                            L3 知识层（Knowledge Layer）                  │
│  向量数据库 │ 关系数据库 │ 缓存层 │ 知识图谱                            │
├─────────────────────────────────────────────────────────────────────────────┤
│                            L4 存储层（Storage Layer）                     │
│  对象存储 │ 版本归档 │ 消息队列 │ 搜索索引                               │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 Agent类型定义

| Agent类型 | 职责 | 输入 | 输出 |
|-----------|------|------|------|
| **调研Agent** | 领域概览、竞品分析、用户需求调研 | 主题 | 结构化调研报告 |
| **写作Agent** | 分章节内容生成 | 素材包+提纲 | 初稿 |
| **检查Agent** | 真实性核查、质量评分 | 初稿 | 检查报告 |
| **合规Agent** | 政治/版权/教育合规检查 | 内容 | 合规报告 |
| **出版Agent** | 排版、图表生成、格式输出 | 定稿 | PDF/DOC |

### 2.3 专家研讨模式

采用**串行论证+并行辩论+角色扮演**混合模式：

```
┌─────────────────────────────────────────────────────────┐
│                   专家研讨流程                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Phase 1: 串行论证（各方独立陈述）                      │
│     ├── 领域专家A陈述观点                              │
│     ├── 领域专家B陈述观点                              │
│     └── 教学法专家陈述观点                              │
│                                                         │
│  Phase 2: 并行辩论（观点交叉）                          │
│     ├── A质疑B的核心假设                               │
│     ├── B质疑C的论据充分性                            │
│     └── C质疑A的实践可行性                            │
│                                                         │
│  Phase 3: 角色扮演（模拟目标读者）                      │
│     ├── 学生视角审阅                                  │
│     ├── 教师视角审阅                                  │
│     └── 出版社编辑视角审阅                             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 3. 核心模块设计

### 3.1 上下文管理机制

采用**分层规划+层次记忆+状态传递**混合架构：

```
┌─────────────────────────────────────────────────────────┐
│                 上下文管理架构                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  L0 全局规划层                                         │
│  └── 全局逻辑链文档（2000-5000字）                      │
│      - 贯穿全书的核心论点                               │
│      - 章节间的依赖关系                                 │
│      - 关键概念定义表                                   │
│      始终保留在上下文中                                  │
│                                                         │
│  L1 章节上下文                                         │
│  └── 当前章节所需的素材摘要                             │
│      - 核心证据（完整摘要）                            │
│      - 支撑证据（令牌引用）                            │
│                                                         │
│  L2 写作上下文                                          │
│  └── 当前段落写作时的实时上下文                         │
│      - 前3段的核心内容                                 │
│      - 已在上下文的引用                                 │
│      - 章节内逻辑链                                    │
│                                                         │
│  L3 状态传递                                           │
│  └── 通过消息队列传递跨章节状态                         │
│      - 已完成章节的核心论点                             │
│      - 关键定义的版本                                   │
│      - 术语表（含已定义/待定义状态）                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 3.2 证据流转机制（混合模式）

```
┌─────────────────────────────────────────────────────────┐
│                 证据流转架构                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  调研Agent输出:                                        │
│  ┌───────────────────┬─────────────────────────────┐  │
│  │   核心证据包      │      支撑证据库             │  │
│  │   (完整摘要)       │      (令牌引用)             │  │
│  ├───────────────────┼─────────────────────────────┤  │
│  │ • 关键事实(3-5条)│ • 次要细节(按需加载)       │  │
│  │ • 核心数据        │ • 历史对比                  │  │
│  │ • 主要论点        │ • 背景信息                 │  │
│  │   (必须完整传递)  │   (令牌化,按需检索)         │  │
│  └───────────────────┴─────────────────────────────┘  │
│                                                         │
│  写作Agent提示词:                                       │
│  "核心证据区包含完整信息,必须直接引用"                  │
│  "支撑证据使用令牌引用格式[@ref:token_id]"            │
│  "禁止在核心证据区进行无引用推断"                       │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

**核心原则**：
- 核心证据（关键事实/核心数据/主要论点）用完整摘要传递
- 支撑证据（次要细节/历史对比）用令牌引用
- 明确标注"必须有证据支持"和"可基于常识推断"的边界

### 3.3 知识图谱与元数据

采用**YAML frontmatter**作为章节元数据声明：

```yaml
# 章节元数据示例
chapter_id: "ch03"
title: "机器学习基础"
dependencies:
  - ch01  # 依赖第1章
  - ch02  # 依赖第2章
follows_to:
  - ch04  # 为第4章铺垫
key_arguments:
  - "监督学习是机器学习的核心范式"
  - "损失函数衡量模型预测与真实值的偏差"
terms_defined:
  - term: "过拟合"
    defined_at: "ch03/section2"
  - term: "正则化"
    defined_at: "ch03/section3"
logic_flow:
  - step: 1
    claim: "机器学习从数据中学习模式"
    evidence_ref: "ev_001"
  - step: 2
    claim: "监督学习需要标注数据"
    evidence_ref: "ev_002"
```

### 3.4 逻辑链文档方案

**设计原则**：逻辑链文档是全书逻辑性的锚点，始终保留在上下文中。

```markdown
# 全局逻辑链文档

## 核心论点链
1. [论点1] 低空经济是新兴战略产业
   - 支撑: 第1章产业定义
   - 支撑: 第2章政策框架

2. [论点2] 技术创新驱动低空经济快速发展
   - 支撑: 第3章技术体系
   - 反驳备选: 若技术成熟度不足,论点需修正

## 章节依赖图
ch01 ─┬─> ch02 ─┬─> ch03
      │        │
      └─> ch04 └─> ch05

## 关键概念定义表
| 概念 | 定义位置 | 首次引用 |
|------|---------|---------|
| 低空经济 | ch01/s1 | - |
| 空域分类 | ch01/s3 | ch02/s1 |
| UTM系统 | ch03/s2 | ch04/s1 |
```

## 4. 提纲生成

### 4.1 四种生成模式

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **全自动AI生成** | 纯AI生成提纲 | 探索性写作、灵感收集 |
| **AI生成+专家审核** | AI生成后由专家审核修订 | 正式项目（推荐） |
| **模板驱动** | 基于已有教材模板填充 | 时间紧迫、格式固定 |
| **用户指定** | 用户提供部分提纲,AI补充完善 | 用户有明确想法 |

### 4.2 提纲结构规范

```yaml
outline:
  metadata:
    title: "人工智能导论"
    author: "AI编写组"
    target_readers: "高等院校计算机专业本科生"
    prerequisite: "高等数学、线性代数、概率论"

  structure:
    - part: "第1篇 基础理论"
      chapters:
        - id: "ch01"
          title: "人工智能概述"
          sections: 3
          key_points: ["AI定义", "发展历程", "研究领域"]
        - id: "ch02"
          title: "机器学习基础"
          sections: 4
          key_points: ["监督学习", "无监督学习", "强化学习"]

    - part: "第2篇 核心技术"
      chapters:
        - id: "ch03"
          title: "深度神经网络"
          sections: 5
          key_points: ["CNN", "RNN", "Transformer"]
```

## 5. 分章节写作

### 5.1 写作流程

```
┌─────────────────────────────────────────────────────────┐
│                 分章节写作流程                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Step 1: 上下文准备                                    │
│     ├── 加载全局逻辑链文档                              │
│     ├── 加载该章节的素材包                              │
│     └── 加载前序章节的核心论点                          │
│                                                         │
│  Step 2: 章节写作                                      │
│     ├── 按节为单位生成                                 │
│     ├── 实时引用核心证据                               │
│     └── 遇到待定义术语查术语表                          │
│                                                         │
│  Step 3: 自检                                          │
│     ├── 检查引用完整性                                 │
│     ├── 检查术语一致性                                 │
│     └── 检查与全局逻辑链的一致性                        │
│                                                         │
│  Step 4: 输出                                          │
│     ├── 初稿                                          │
│     ├── 章节元数据                                     │
│     └── 待验证引用列表                                 │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 写作质量标准

| 指标 | 目标值 | 检测方式 |
|------|--------|---------|
| 字数偏差 | ±10% | 自动统计 |
| 引用覆盖率 | 100%（核心主张） | 引用检测 |
| 术语一致性 | 100% | 术语表对照 |
| 逻辑连贯性 | ≥0.8 | LLM评分 |

## 6. 人工介入节点

### 6.1 强制人工介入节点

| 节点 | 类型 | 说明 |
|------|------|------|
| **提纲审核** | HARD BLOCK | 必须通过才能进入写作 |
| **关键章节初审** | SOFT GATE | 不通过则重写,不阻塞其他章节 |
| **终审** | HARD BLOCK | 必须通过才能发布 |

### 6.2 专家审核职责

**提纲审核时专家判断**：
- "这本书为什么应该存在"——独特价值主张
- 受众适配性——深度是否合适
- 教学法合理性——概念引入顺序
- 内容完整性——是否遗漏核心内容

**关键章节初审时专家判断**：
- 方向性是否正确
- 概念定义是否与领域主流一致
- 是否有"AI盲区"（情感一致性、文化隐喻等）

### 6.3 反馈格式规范

```yaml
review_feedback:
  type: "outline_review"  # outline_review | chapter_review | final_review
  version: "v2.1"
  reviewer: "张教授"
  date: "2026-05-18"

  decision: "revision_required"  # approved | revision_required | rejected

  issues:
    - id: "P1-1"
      priority: "critical"
      location: "第3章/整体框架"
      description: "机器学习章节顺序不合理,应先讲监督学习"
      suggestion: "将3.1和3.2顺序对调"
      status: "open"  # open | resolved | accepted_no_change

    - id: "P2-1"
      priority: "high"
      location: "第1章/section2"
      description: "案例过于陈旧"
      suggestion: "更新为2024年案例"
      status: "open"
```

## 7. 真实性检查

### 7.1 检查架构

```
┌─────────────────────────────────────────────────────────┐
│               真实性检查架构（AI监督AI）                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Tier 1: 数值/法规类 ──→ 外部API核实 ──→ 极高置信度   │
│  └── 调用官方数据库核实                                 │
│                                                         │
│  Tier 2: 引用类     ──→ DOI/CrossRef ──→ 高置信度      │
│  └── 验证文献DOI是否存在                               │
│                                                         │
│  Tier 3: 概念/论述  ──→ AI检查 ──→ 低置信度 → 人工复核 │
│  └── 检查逻辑一致性                                     │
│                                                         │
│  Tier 4: 高风险章节 ──→ 人工深度审核                    │
│  └── 专家逐字审读                                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 7.2 幻觉分类与防御

| 幻觉类型 | 风险等级 | 检测方法 | 防御策略 |
|---------|---------|---------|---------|
| **事实性幻觉** | 高 | 外部API核实 | 数据必须来自可信源 |
| **引用幻觉** | 极高 | DOI解析+CrossRef | 强制DOI解析,无DOI不引用 |
| **概念性幻觉** | 中 | 专家定义表核对 | 术语必须查表 |
| **数字性幻觉** | 高 | 统计数据库核实 | 数据必须标注来源 |

### 7.3 红队审查机制

```yaml
red_team:
  purpose: "主动注入错误,验证检查Agent能力"

  injection_types:
    - type: "factual_error"
      injection_rate: 0.1
      example: "将"2023年"改为"2024年"（实际是2022年）"

    - type: "citation_fabrication"
      injection_rate: 0.05
      example: "添加不存在的DOI"

    - type: "conceptual_confusion"
      injection_rate: 0.1
      example: "混淆"准确率"和"精确率"定义"

  evaluation_metrics:
    - detection_rate: "应>95%"
    - false_positive_rate: "应<10%"
```

## 8. 合规检查框架

```
┌─────────────────────────────────────────────────────────┐
│                  五级合规管理体系                        │
├─────────────────────────────────────────────────────────┤
│ L5 教育政策层                                          │
│     └─ 教育部规章制度、马工程重点教材                   │
├─────────────────────────────────────────────────────────┤
│ L4 教学标准层                                          │
│     └─ 课程标准、专业教学标准、实训标准                 │
├─────────────────────────────────────────────────────────┤
│ L3 内容规范层                                          │
│     ├─ 政治合规（表述禁区+必含内容）                    │
│     ├─ 科学合规（知识准确性+数据时效性）                │
│     ├─ 教学合规（体系完整性+能力对应）                  │
│     └─ 格式合规（引用规范+版权合规）                   │
├─────────────────────────────────────────────────────────┤
│ L2 检查执行层                                          │
│     ├─ 前置检查：任务定义→边界预设→监控阈值            │
│     ├─ 过程检查：实时过滤+偏离预警+强制中断             │
│     └─ 后置检查：覆盖率+一致性+多级审核                │
├─────────────────────────────────────────────────────────┤
│ L1 技术支撑层                                          │
│     ├─ 政策知识库（实时更新API）                       │
│     ├─ 课程标准库（版本化管理）                        │
│     ├─ 合规规则引擎（可配置阈值）                     │
│     └─ 追溯日志系统（全流程记录）                      │
└─────────────────────────────────────────────────────────┘
```

### 8.2 政治合规检查

**强制人工审核内容**：
- 领土主权（台/港/藏/疆/南海）
- 党史国史敏感事件
- 领导人相关表述
- 民族宗教政策

**AI辅助检测**：敏感词/禁用词扫描，但**永远不能替代人工审核**

### 8.3 版权合规

| 引用类型 | 阈值 | 处理方式 |
|---------|------|---------|
| 直接引用 | <100字或<10% | 需标注来源 |
| 整体相似度 | <15% | 🟢通过 |
| 整体相似度 | 15-30% | 🟡需审核 |
| 整体相似度 | >30% | 🔴需改写 |

## 9. 排版与输出

### 9.1 内容与样式分离原则

```
AI生成内容（语义标签）
        ↓
样式映射层（模板引擎）
        ↓
┌───────┴───────┐
↓               ↓
PDF输出         DOC输出
```

### 9.2 排版规范

| 元素 | 规范 |
|------|------|
| 开本 | 大16开(220×305mm) |
| 正文字体 | 宋体/思源宋体 10.5pt |
| 标题层级 | 篇(18pt)→章(16pt)→节(14pt)→小节(12pt) |
| 行距 | 1.5倍行距 |
| 页边距 | 上35mm,下30mm,左30mm,右25mm |

## 11. 安全体系

### 11.1 安全架构原则

- **零信任**：不信任任何输入,所有内容需验证
- **最小权限**：Agent只有完成任务的最小必要权限
- **深度防御**：多层安全防护,单点失效不影响全局
- **沙箱隔离**：外部内容提取在沙箱中执行

## 16. 遗漏项与实施路线图

### 16.1 优先级P0（系统无法运行的基础缺失）

| 遗漏项 | 说明 | 建议方案 |
|--------|------|---------|
| 流程引擎 | 缺少状态机定义 | Temporal/Airflow |
| 消息队列 | Agent间通信机制 | Kafka/Redis Streams |
| 错误处理 | 无重试/回滚/熔断 | SAGA模式 |
| 分布式追踪 | 无法串联全链路 | OpenTelemetry |
| 密钥管理 | API密钥明文存储 | Vault |
| Prompt注入防御 | 恶意指令可能劫持Agent | 输入验证+输出过滤 |
| CI/CD | 代码变更无法自动验证 | GitHub Actions |

### 16.2 优先级P1（系统可靠性不足）

| 遗漏项 | 说明 |
|--------|------|
| 版本控制 | 素材/提纲/初稿的版本管理 |
| 回滚机制 | 检查点快照+SAGA |
| 人工介入集成 | 任务队列+Webhook |
| 监控仪表盘 | Grafana |
| 测试体系 | 完成度约8-10% |
| 向量数据库选型 | PgVector vs Milvus |
| 数据管线 | 采集→切片→嵌入→存储 |
| Golden Dataset | 参考基准 |
