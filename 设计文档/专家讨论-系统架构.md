# AI教材编写系统 - 系统架构设计

> 版本：v1.0  
> 日期：2025-05-18  
> 角色：系统架构专家（分布式系统 × 知识管理 × 上下文建模）

## 一、核心约束与假设

| 参数 | 数值 | 说明 |
|------|------|------|
| 教材总规模 | 20万字 / ~100万tokens | 约20章 × 1万字/章 |
| 模型上下文窗口 | 200K tokens | 约8万字 |
| 单章规模 | ~1万字 | 远小于上下文窗口 |
| 单章素材包 | ~2-3万字 | 参考文献、案例、数据 |
| 全局逻辑链 | ~1-2万字 | 核心概念、术语表、论证链 |

**关键观察**：单章1万字远小于200K上下文，但这不代表单章写作可以不加节制。真正的瓶颈在于**累积上下文**——全局逻辑链 + 素材包 + 前序章节内容的叠加。

## 二、上下文窗口设计

### 2.1 单章写作时的上下文组成

```
┌─────────────────────────────────────────────────────────────┐
│                    200K 上下文窗口预算                        │
├──────────────────┬──────────────────┬───────────────────────┤
│   全局逻辑链      │    素材包         │    前序章节摘要         │
│   (~15K tokens)  │   (~40K tokens)  │   (~30K tokens)        │
├──────────────────┴──────────────────┴───────────────────────┤
│                    当前章节草稿                                │
│                    (~10K tokens)                              │
├─────────────────────────────────────────────────────────────┤
│                    模型响应空间                                │
│                    (~100K tokens)                             │
└─────────────────────────────────────────────────────────────┘
```

**分配建议**：

| 区块 | tokens | 占比 | 说明 |
|------|--------|------|------|
| 全局逻辑链 | 15K | 7.5% | 核心概念定义、术语表、论证主线 |
| 素材包 | 40K | 20% | 参考文献、案例、数据（可切分） |
| 前序章节摘要 | 30K | 15% | 仅关键论点、结论、引用点 |
| 当前草稿 | 10K | 5% | 正在写作的章节 |
| 模型响应 | 100K | 50% | 留给模型输出的空间 |

### 2.2 上下文预算管理策略

#### 策略A：固定预算池（Recommended）

```
每个会话开始时分配固定预算：

Session Budget = {
  global_logic: 15K,      // 不可压缩
  material_pool: 40K,     // 可按需替换
  prev_chapters: 30K,    // 动态压缩
  draft: 10K,            // 实时增长
  response: 100K          // 固定保留
}
```

**优势**：预算明确，不会出现突发性OOM
**劣势**：灵活性较低，素材替换需要手动管理

#### 策略B：优先级驱逐（Priority Eviction）

```
优先级队列（从低到高）：
1. 素材包（可按需加载）
2. 前序章节全文
3. 前序章节摘要（压缩版）
4. 全局逻辑链（始终保留）
```

**优势**：自动适应不同章节的复杂度
**劣势**：驱逐行为不可预测，可能丢失重要上下文

**推荐**：采用策略A的变体——**固定核心 + 弹性素材**

```
Global Budget = {
  CORE (60K, 不可压缩):
    - 全局逻辑链: 15K
    - 前序章节摘要: 30K
    - 当前草稿+响应: 15K

  MATERIAL (可变, 上限40K):
    - 素材包（按需加载）

  RESERVE (100K, 始终保留):
    - 模型响应空间
}
```

### 2.3 多轮对话的上下文管理

#### 问题：多轮累积导致上下文膨胀

```
Round 1: 消耗 50K tokens
Round 2: 对话历史 +20K → 70K
Round 3: 对话历史 +20K → 90K
...
Round N: 逐渐逼近预算上限
```

#### 方案：对话快照压缩（Session Snapshot）

```
每个"写作阶段"结束时执行快照：

Snapshot = {
  chapter_draft: "当前章节完整草稿",
  referenced_logic: "本阶段使用的逻辑链片段",
  material_manifest: "素材包清单（不含原文）",
  decision_log: "本阶段关键决策（为什么这样写）"
}

下一阶段开始时：
1. 加载 Snapshot
2. 仅当需要时按需加载素材原文
3. 对话历史仅保留最近3轮
```

#### 会话分层架构

```
┌─────────────────────────────────────────────┐
│           会话层 (Session)                    │
│  - 对话历史（最近3轮）                         │
│  - 当前阶段目标                              │
│  - 运行时栈（局部变量）                        │
├─────────────────────────────────────────────┤
│           阶段层 (Phase)                     │
│  - 快照（章节级）                             │
│  - 素材包指针                                │
│  - 逻辑链版本号                              │
├─────────────────────────────────────────────┤
│           项目层 (Project)                   │
│  - 全局逻辑链                                │
│  - 术语表                                   │
│  - 章节索引                                  │
│  - 依赖图                                   │
└─────────────────────────────────────────────┘
```

## 三、跨章节逻辑一致性方案

### 3.1 逻辑链文档（Logic Chain Document）的定位

**核心定位**：逻辑链文档是**跨越章节的推理骨架**，不是章节大纲。

```
逻辑链文档结构：

# 全局逻辑链 v1.3
## 核心概念定义
- 概念A: 定义 + 首次引入章节 + 引用计数
- 概念B: 定义 + 首次引入章节 + 引用计数
...

## 论证主线
- 主线1: 论点 → 论据(章) → 推论
- 主线2: 论点 → 论据(章) → 推论
...

## 章节依赖图
ch01 ─┬─→ ch03 ─┬─→ ch05
      └─→ ch04 ─┴─→ ch07
ch02 ─────────────────→ ch06

## 术语一致性表
| 术语 | 标准定义 | 禁用变体 | 引入章节 |
|------|----------|----------|----------|
| XXX  | ...      | ...      | ch01     |
```

### 3.2 章节依赖关系的维护机制

#### 依赖类型建模

```typescript
type DependencyType =
  | 'concept'      // 概念依赖（需要先引入XXX概念）
  | 'argument'     // 论据依赖（需要用到ch03的数据）
  | 'reference'    // 引用依赖（需要引用ch05的结论）
  | 'terminology'  // 术语依赖（需要遵循ch02的定义）

interface ChapterDependency {
  source: string           // e.g., "ch03"
  target: string           // e.g., "ch07"
  type: DependencyType
  weight: number           // 依赖强度 0-1
  details: {
    concept?: string       // 具体概念名
    section?: string       // 具体章节/段落
    rationale?: string     // 依赖原因说明
  }
}
```

#### 依赖验证机制

```
写作时检查流程：

1. 章节开始写作前
   ├─→ 检查前置依赖是否已通过审核
   ├─→ 若依赖未通过，阻塞写作并提示
   └─→ 生成"依赖检查报告"

2. 章节写作完成后
   ├─→ 验证所有声明的引用是否存在
   ├─→ 检查是否有未声明的隐含依赖
   └─→ 更新依赖图状态
```

#### 依赖冲突检测

```
场景：ch07声称"根据ch03的定义"，但ch03的定义已被修订

检测流程：
1. 每次ch03修订时，扫描所有后续章节
2. 标记所有依赖ch03定义的章节
3. 生成"影响分析报告"
4. 可选：自动触发ch07的修订检查
```

### 3.3 术语一致性的技术实现

#### 术语表服务（Terminology Service）

```typescript
interface TerminologyEntry {
  term: string
  canonical_form: string           // 标准化形式
  definition: string
  scope: 'global' | 'chapter'     // 适用范围
  introduced_in: string            // 首次引入位置
  variants: string[]              // 同义词/禁用形式
  usage_guidelines: string         // 使用说明
  last_verified: string           // 最后验证时间
  status: 'stable' | 'proposed' | 'deprecated'
}

interface TerminologyService {
  // 注册新术语
  register(entry: TerminologyEntry): Promise<void>

  // 查询术语
  lookup(term: string): Promise<TerminologyEntry | null>

  // 更新术语
  update(term: string, patch: Partial<TerminologyEntry>): Promise<void>

  // 冲突检测
  detect_conflicts(chapter_draft: string): ConflictReport

  // 批量验证
  validate_chapter(chapter_id: string): ValidationResult
}
```

#### 写作时的术语检查

```
实时检查流程：

1. 术语识别
   ├─→ 扫描新写入的文本
   ├─→ 识别可能的术语使用

2. 三种处理模式
   ├─→ GREEN（一致）: 术语使用符合术语表
   ├─→ YELLOW（警告）: 术语使用存在变体/同义形式
   └─→ RED（错误）: 术语使用与定义矛盾

3. 用户反馈
   ├─→ YELLOW: 提示是否使用标准形式
   └─→ RED: 阻塞写作，要求确认是否修订术语表
```

### 3.4 逻辑链版本管理

```
版本策略：语义版本 + 变更日志

v1.3.2
├─ major: 1  (核心概念定义发生根本性变化)
├─ minor: 3  (新增章节/论证链)
└─ patch: 2  (措辞/表述调整)

变更日志：
[ch03] 修订"XXX"定义 v1.2 → v1.3
├─ 原因: 与最新行业标准对齐
├─ 影响章节: ch05, ch07, ch10
├─ 影响评估: 需要复核共3处引用
└─ 状态: 等待ch05确认
```

## 四、引用架构设计

### 4.1 三层引用的处理差异

```
┌─────────────────────────────────────────────────────────────┐
│                    三层引用模型                              │
├─────────────────┬─────────────────┬─────────────────────────┤
│     章内引用     │     跨章引用      │      外部引用            │
│   (Intra)       │    (Inter)      │      (External)         │
├─────────────────┼─────────────────┼─────────────────────────┤
│ 范围: 本章内     │ 范围: 跨章节     │ 范围: 教材外来源          │
│ 稳定性: 高       │ 稳定性: 中       │ 稳定性: 依赖外部资源       │
│ 解析: 写作时解析   │ 解析: 写作后解析  │ 解析: 写作时声明         │
│ Token消耗: 低    │ Token消耗: 中    │ Token消耗: 高（需加载原文） │
└─────────────────┴─────────────────┴─────────────────────────┘
```

#### 章内引用（Intra-Chapter）

**特点**：
- 引用本章内的图表、案例、数据
- 上下文稳定，不受外部变化影响
- 可在写作时直接解析

**处理策略**：
```
写作时：
  "如图3.2所示" → 解析为 "如图{本章编号}.{本章内序号}所示"

优势：
  - 实时解析，无延迟
  - 无额外API调用
  - 引用稳定，不会产生"孤儿引用"
```

#### 跨章引用（Inter-Chapter）

**特点**：
- 引用其他章节的概念、结论、数据
- 依赖目标章节的稳定性
- 需要维护引用关系图

**处理策略**：
```
声明式引用（Recommended）：

写作时使用标记：
  "[[ch05:概念A]]"  ← 表示引用ch05中的概念A
  "[[ch07:结论]]"   ← 表示引用ch07的结论

写作后解析：
  系统扫描所有 [[chXX:yyy]] 标记
  替换为正式引用格式：
    "根据第5章对'概念A'的定义..."
    "如图7.3所示..."

优势：
  - 写作时无需加载目标章节内容
  - 便于检测悬空引用（目标不存在）
  - 章节编号变化时可批量更新
```

#### 外部引用（External）

**特点**：
- 引用外部文献、数据、案例
- 依赖外部资源的稳定性
- Token消耗高（需要加载原文）

**处理策略**：
```
外部引用声明：

写作时：
  "@@文献ID:观点摘要@@"  ← 声明引用，不嵌入原文

素材包管理：
  外部文献存储在素材包中
  写作时仅加载观点摘要（~200字）
  完整原文按需加载

优势：
  - 控制Token消耗
  - 保留溯源能力
  - 外部资源变更时可警报
```

### 4.2 引用解析时机选择

#### 方案对比

| 时机 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| 写作时解析 | 实时反馈，写作流畅 | 需要目标内容在上下文 | 章内引用 |
| 写作后解析 | 不阻塞写作，灵活性高 | 可能遗漏错误 | 跨章引用 |
| 编译时解析 | 统一处理，全局优化 | 反馈延迟长 | 终稿审核 |

**推荐：混合策略**

```
┌─────────────────────────────────────────┐
│              引用解析流水线               │
├─────────────────────────────────────────┤
│ 1. 写作时                                │
│    ├─→ 章内引用：实时解析                 │
│    └─→ 跨章引用：标记 + 警告（不阻塞）      │
│                                         │
│ 2. 阶段结束时                            │
│    ├─→ 跨章引用：批量解析                 │
│    └─→ 悬空引用检测                       │
│                                         │
│ 3. 编译时                                │
│    ├─→ 全局引用一致性验证                 │
│    ├─→ 格式统一化                        │
│    └─→ 生成引用索引                       │
└─────────────────────────────────────────┘
```

### 4.3 引用稳定性保障

#### 引用令牌（Reference Token）

```
核心问题：当上下文压缩/切片时，引用token是否也会变化？

传统方案（有问题）：
  "根据第5章的概念A定义..." → 上下文压缩后 → "根据第5章..."

推荐方案：引用令牌

  写作时：[[ch05:concept_a]]
  解析后：ch05_ref_concept_a_20250518  ← 稳定令牌

优势：
  - 令牌不随上下文变化
  - 便于全局索引和更新
  - 支持反向查询（某引用被哪些章节使用）
```

#### 引用版本锚定

```
场景：ch05在v1.3时引入了概念A，但ch07在v1.5时才引用

引用记录：
  {
    source: "ch07",
    target: "ch05",
    target_id: "concept_a",
    target_version: "v1.3",
   引用引入版本: "v1.5",
    状态: "verified"  // stable | modified | broken
  }

当ch05的概念A发生修订时：
  1. 检测所有依赖ch05概念A的章节
  2. 标记受影响引用
  3. 提示需要复核的章节
```

## 六、权衡分析与潜在风险

### 6.1 上下文窗口设计

| 权衡点 | 选择 | 理由 | 潜在风险 |
|--------|------|------|----------|
| 预算分配 | 固定核心 + 弹性素材 | 核心内容稳定，素材按需加载 | 素材切换时可能丢失重要上下文 |
| 对话管理 | 快照 + 3轮历史 | 平衡内存与信息保留 | 早期决策追溯困难 |
| 多章并行 | 不支持（强串行） | 保证逻辑一致性 | 效率受限，但质量更可控 |

### 6.2 逻辑一致性

| 权衡点 | 选择 | 理由 | 潜在风险 |
|--------|------|------|----------|
| 逻辑链粒度 | 概念级（非段落级） | 避免过度耦合 | 细节论证的一致性依赖人工 |
| 依赖验证 | 写作前检查（硬阻塞） | 防止无效写作 | 灵活性和效率 |
| 术语冲突处理 | 写作时警告（不阻塞） | 避免打断思路 | 依赖作者自律 |

### 6.3 引用架构

| 权衡点 | 选择 | 理由 | 潜在风险 |
|--------|------|------|----------|
| 跨章引用解析 | 写作后解析 | 不阻塞写作思维 | 错误发现延迟 |
| 外部引用 | 摘要先行 | 控制Token消耗 | 引用不准确时难以发现 |
| 引用令牌 | 强制使用 | 全局一致性和可追溯性 | 增加学习成本 |

### 6.4 关键风险项

```
高风险：
1. 逻辑链文档与实际写作产生偏差 → 建立定期同步机制
2. 跨章引用悬空（目标章节被大幅修订） → 建立影响分析预警
3. 素材包过大导致加载超时 → 实现智能预加载策略

中风险：
4. 多轮对话后上下文质量下降 → 定期强制快照重建
5. 术语表更新后历史章节出现不一致 → 记录版本快照
6. 外部引用资源失效 → 建立外部资源监控
```

## 七、关键决策建议

### 决策1：上下文预算分配

**推荐方案**：固定核心60K + 弹性素材40K + 响应保留100K

**理由**：
- 核心内容（逻辑链 + 前序摘要）是稳定成本，应固定
- 素材是按需成本，应弹性管理
- 100K响应空间是模型质量的保障，不能压缩

### 决策2：跨章引用时机

**推荐方案**：声明式引用 + 写作后批量解析

**理由**：
- 写作时声明引用，不阻塞思维
- 写作后批量解析，可以全局优化
- 配合版本锚定，支持历史追溯

### 决策3：逻辑链粒度

**推荐方案**：概念级 + 关键论证链（不深入段落级）

**理由**：
- 段落级逻辑链维护成本过高
- 概念级足以保证跨章节一致性
- 细节一致性通过人工审核保障
