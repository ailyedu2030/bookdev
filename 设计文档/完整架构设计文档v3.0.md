## 第一部分：系统概述

### 1.1 项目背景

本项目旨在构建一个基于AI多智能体协同的教材编写系统，目标规模为**20万汉字**（约100万Tokens），服务于商学院企业管理、市场营销、人力资源管理等专业的教材开发。

**核心挑战**：
- **规模复杂性**：20万字教材需多轮迭代，涉及大纲生成、内容编写、审核修订的完整流程
- **质量一致性**：需保证AI生成内容的专业性、准确性和风格一致性
- **知识规范性**：教材内容需符合中国出版行业规范，满足CIP数据申报要求
- **成本可控性**：需在合理Token预算内完成，避免上下文膨胀导致的额外消耗
- **幻觉控制**：LLM在长文本场景下幻觉概率显著增加，引用类幻觉检测率<40%
- **逻辑一致性**：切片方案会导致章节间逻辑性丢失

### 1.2 目标模型约束

| 参数 | 值 |
|------|-----|
| **目标模型** | MiniMax-M2.7 |
| **上下文窗口** | 200K Tokens（约8万字） |
| **输入成本** | ¥2/1M tokens |
| **输出成本** | ¥4/1M tokens |
| **设计基准** | 32K-64K（保守估算） |

### 1.3 系统目标

| 目标维度 | 具体指标 |
|----------|----------|
| **规模目标** | 支持20万汉字（约100万Tokens）规模教材编写 |
| **质量目标** | 术语一致率≥95%、知识准确率≥98%、格式规范率≥99% |
| **效率目标** | 大纲生成<30分钟/章、内容生成<2小时/节、审核<30分钟/节 |
| **成本目标** | Token消耗≤150万/册 |
| **合规目标** | 重大选题备案率100%、CIP数据完整率100%、原创性相似度<15% |
| **幻觉检出** | 数值/法规类≥95%、引用类≥90%、概念类≥70% |

### 1.4 设计原则

1. **分层解耦**：将系统划分为规划层、执行层、认知层、工具层，各层职责明确
2. **状态驱动**：以知识图谱为状态核心，通过边结构维护章节间的逻辑关联
3. **过程可观测**：引入Temporal作为流程引擎，确保每个环节可追溯、可审查
4. **预算友好**：核心上下文60K + 弹性材料40K的双层架构，控制Token消耗
5. **合规内建**：将出版规范嵌入工作流节点，而非事后检查
6. **安全优先**：纵深防御体系，LLM调用必须带版本戳，不可变日志

### 2.2 技术栈总览

| 层级 | 技术选型 | 核心职责 |
|------|----------|----------|
| **大模型** | MiniMax-M2.7 (200K context, ¥2/1M tokens) | 内容生成，理解推理，多轮对话 |
| **流程引擎** | Temporal + Redis Streams | 工作流编排，状态管理，断点续写 |
| **知识存储** | PostgreSQL + apoc扩展 | 知识图谱存储，关系数据持久化 |
| **向量检索** | Qdrant / Milvus 3.0 | 语义检索，RAG上下文获取 |
| **缓存层** | Redis Cluster | Token预算管理，热点数据缓存 |
| **消息队列** | Kafka | 异步事件分发，日志聚合 |
| **前端** | React + Ant Design Pro | 编写界面，进度监控，结果展示 |
| **后端** | Python/FastAPI | API服务，业务逻辑，LLM封装 |
| **密钥管理** | HashiCorp Vault | 密钥轮换，审计日志 |

### 2.3 核心架构决策

#### 决策1：流程引擎选型 - Temporal + LangGraph混合

| 方案 | 优势 | 劣势 | 适用场景 |
|------|------|------|----------|
| LangChain Agents | 生态成熟，集成度高 | 长时间运行不稳定 | 快速原型，POC |
| Temporal | 状态持久化好，断点续写原生支持 | 学习曲线陡，运维成本高 | 生产级长时间工作流 |
| **Temporal + LangGraph** | 互补：Tempora管工作流，LangGraph管Agent协作 | 架构复杂 | **采用** |

#### 决策2：Token预算架构

| 层级 | 预算 | 内容 |
|------|------|------|
| **L0 核心上下文** | 60K Tokens | 全局逻辑链（大纲架构、术语表、风格指南、角色设定） |
| **L1 弹性材料** | 40K Tokens | 当前章节的参考内容，RAG检索结果 |
| **L2 章节状态** | ~5K Tokens/章 | 章节元数据（进度、版本、审核状态） |
| **L3 输出缓冲** | 10K Tokens | 响应正文生成空间 |
| **总计** | **~115K Tokens/章** | 预留15%余量 |

#### 决策3：位置引用改为内容寻址哈希

```python
内容寻址哈希 = SHA-256(内容片段文本)
引用格式 = {hash: "abc123...", offset: 0, length: 500}
```

优势：位置无关、自动去重、可验证完整性

| 节点类型 | 属性 | 说明 |
|----------|------|------|
| **Chapter** | id, title, order, status, wordCount, version | 章节点 |
| **Section** | id, title, order, status, wordCount, parentChapterId | 节节点 |
| **Subsection** | id, title, order, status, content, parentSectionId | 小节节点 |

### 3.2 边设计

| 边类型 | 起点 | 终点 | 属性 |
|--------|------|------|------|
| **CONTAINS** | Chapter | Section | order |
| **FOLLOWS** | Chapter/Section | Chapter/Section | logicalType∈{前置,后续,可选} |
| **DEFINES** | Chapter/Section | Concept | definitionText |
| **USES** | Chapter/Section | Term | usageContext |
| **REFERENCES** | Section | Section | referenceType, context |
| **ASSIGNED_TO** | Chapter/Section | Author | assignedAt |
| **REVIEWED_BY** | Chapter/Section | Author | reviewAt, status |
| **SIMILAR_TO** | Concept | Concept | similarityScore |

## 第四部分：多Agent协同架构

### 4.1 Agent类型定义

| Agent | 角色 | 输入 | 输出 | 协作方式 |
|-------|------|------|------|----------|
| **大纲生成Agent** | 结构设计师 | 学科信息，培养方案 | 教材大纲 | 触发章节生成 |
| **内容生成Agent** | 主笔 | 章节要求，上下文，风格指南 | 章节草稿 | 触发审核 |
| **审核修订Agent** | 质量把关 | 章节草稿，审核标准 | 审核意见+修订稿 | 通知生成Agent |
| **格式编排Agent** | 排版工程师 | 内容+格式规范 | 格式化文档 | 触发出版检查 |
| **术语管理Agent** | 术语官 | 全局术语表，新术语 | 术语审核意见 | 通知生成Agent |
| **进度管控Agent** | 项目经理 | 任务状态，资源消耗 | 进度报告 | 协调各Agent |

### 4.2 人工介入节点

| 节点 | 类型 | 说明 |
|------|------|------|
| **提纲审核** | HARD BLOCK | 必须通过才能进入写作 |
| **关键章节初审** | SOFT GATE | 不通过则重写，不阻塞其他章节 |
| **终审** | HARD BLOCK | 必须通过才能发布 |

### 4.3 超时与重试策略

| 场景 | 超时时间 | 重试策略 | 降级方案 |
|------|----------|----------|----------|
| 内容生成 | 5分钟/节 | 指数退避：1min→2min→4min | 返回上版草稿，标记需人工审核 |
| 术语检查 | 30秒 | 1次重试 | 跳过检查，生成警告日志 |
| 审核修订 | 2分钟/节 | 不重试 | 人工介入 |
| 图谱更新 | 10秒 | 3次重试 | 写入本地队列，异步重试 |

## 第五部分：上下文管理架构

### 5.1 分层策略

```
┌─────────────────────────────────────────────────────────┐
│                 上下文管理架构                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  L0 全局规划层（必须保留）                             │
│  └── 全局逻辑链文档（~10K）                            │
│      - 贯穿全书的核心论点                               │
│      - 章节间的依赖关系                                 │
│      - 关键概念定义表                                   │
│      始终保留在上下文中                                  │
│                                                         │
│  L1 章节上下文（可压缩）                               │
│  └── 当前章节所需的素材摘要                             │
│      - 核心证据（完整摘要）                            │
│      - 支撑证据（令牌引用）                            │
│                                                         │
│  L2 写作上下文（实时）                                 │
│  └── 当前段落写作时的实时上下文                         │
│      - 前3段的核心内容                                 │
│      - 已在上下文的引用                                 │
│      - 章节内逻辑链                                    │
│                                                         │
│  L3 状态传递（外部）                                   │
│  └── 通过消息队列传递跨章节状态                         │
│      - 已完成章节的核心论点                             │
│      - 关键定义的版本                                   │
│      - 术语表（含已定义/待定义状态）                    │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 5.2 上下文预算控制原则

1. L0内容固定只读，仅在项目初始化和重大变更时更新
2. L1内容动态加载，每章节编写前根据内容寻址哈希获取
3. L2状态轻量存储，不计入上下文预算
4. 单章节超出预算时，触发RAG检索淘汰最远章节

### 5.3 上下文膨胀控制

**触发条件**：单章节上下文超过45K Tokens时

**控制策略**：
1. **RAG淘汰**：优先淘汰最远章节的参考内容
2. **摘要压缩**：对L1材料进行摘要，保留关键信息
3. **历史截断**：仅保留最近3轮的对话历史

## 第六部分：幻觉检测与规避架构

### 6.1 幻觉分类与检测策略

| 类型 | 描述 | 检测策略 | 规避方法 |
|------|------|----------|----------|
| **事实性幻觉** | 与可验证事实矛盾 | 知识库比对，RAG验证 | 引入外部知识检索 |
| **引用幻觉** | 捏造不存在的引用 | DOI/CrossRef API验证 | 强制DOI解析，无DOI不引用 |
| **概念幻觉** | 概念定义在后续章节被悄然修改 | 术语表一致性检查 | 锁定首次定义，后续引用必须一致 |
| **数值幻觉** | 捏造统计数据 | 外部API核实 | 数据必须标注来源 |

### 6.2 四层检测体系

```
┌─────────────────────────────────────────────────────────┐
│               真实性检查架构（AI监督AI）                  │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  Tier 1: 数值/法规类 ──→ 外部API核实 ──→ 极高置信度   │
│  └── 调用官方数据库核实                                 │
│                                                         │
│  Tier 2: 引用类     ──→ DOI/CrossRef ──→ 高置信度    │
│  └── 验证文献DOI是否存在                               │
│                                                         │
│  Tier 3: 概念/论述  ──→ AI检查 ──→ 低置信度 → 人工复核 │
│  └── 检查逻辑一致性                                     │
│                                                         │
│  Tier 4: 高风险章节 ──→ 人工深度审核                   │
│  └── 专家逐字审读                                     │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 6.3 引用验证流程

```
1. 写作Agent生成内容，含引用[@ref:xxx]
2. 检查Agent提取所有DOI令牌
3. 调用CrossRef API验证DOI存在性
4. 如DOI不存在 → 标记为P0缺陷，要求重写
5. 如DOI存在 → 验证引用内容相关性
6. 高风险引用（法规/数据）→ 二次外部API核实
```

### 7.2 政治合规

**强制人工审核内容**：
- 领土主权（台/港/藏/疆/南海）
- 党史国史敏感事件
- 领导人相关表述
- 民族宗教政策

**AI辅助检测**：敏感词/禁用词扫描，但**永远不能替代人工审核**

### 7.3 版权合规

| 引用类型 | 阈值 | 处理方式 |
|---------|------|---------|

| 连续引用 | >300字符 | 必须获授权 |
| 整体相似度 | <15% | 🟢通过 |
| 整体相似度 | 15-30% | 🟡需审核 |
| 整体相似度 | >30% | 🔴需改写 |

## 第八部分：排版与输出

### 8.1 内容与样式分离原则

        ↓

        ↓
┌───────┴───────┐

PDF输出         DOC输出
