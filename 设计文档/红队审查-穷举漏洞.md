# 红队审查报告：AI多Agent教材编写系统

**审查日期**：2026-05-18  
**审查范围**：LLM幻觉漏洞、上下文管理漏洞，知识图谱攻击，工作流攻击、合规性绕过  
**假设前提**：系统尚未实现，基于架构设计进行威胁建模

## B. 每个漏洞的利用路径

### H-001: 引用幻觉自我强化循环

**发现方式**：
```
1. 检查Agent在第3章发现错误引用[@ref:xxx]
2. 但写作Agent在第5章引用了第3章的内容（含错误引用）
3. 检查Agent验证第5章时，溯源到第3章的原始引用
4. 由于第3章已被标记为"历史版本"，检查Agent认为已核实
5. 错误引用通过"历史版本溯源"被间接验证
```

**利用条件**：
- 版本戳机制未正确实现
- 检查Agent使用"来源章节状态"而非"引用事实状态"判断可信度

**攻击路径**：
```
[写入错误引用] → [被后续章节引用] → [版本固化] → [溯源时跳过核实] → [错误固化]
```

---

### H-002: 跨章节级联幻觉传播

**发现方式**：
```
1. 攻击者在调研素材中植入"2024年数据：中国AI市场规模达5000亿元"
2. 调研Agent将此素材加入素材包（未核实）
3. 写作Agent在第2章使用此数据作为市场背景
4. 第5章引用："正如第2章所述，中国AI市场规模达5000亿元"
5. 第8章进一步引用："根据第2章和第5章的数据..."
6. 错误数据已在全书中形成"证据链"
```

**利用条件**：
- 素材来源核实机制缺失
- 数值类事实未强制通过外部API核实
- 跨章引用未做二次核实

---

### H-003: 数值幻觉的指数放大

**发现方式**：
```
1. 初始错误：写作Agent生成"根据xxx报告，2024年增长率达30%"
2. 第一次放大：其他章节引用此数据，并添加"其中xxx贡献了50%"
3. 第二次放大：后续章节计算"30% × 50% = 15%的增长"
4. 第三次放大：结论章节使用"综合计算，年均增长15%"
5. 原始误差±5%，经过3次传播放大为±15%
```

**利用条件**：
- 数值传播路径未追踪
- 中间计算结果未做回溯验证
- 缺乏"数值传播图谱"

---

### H-004: 法规条文幻觉合成

**发现方式**：
```
1. 攻击者构建素材："根据《人工智能法》第28条规定..."
2. 写作Agent在正文中引用："依据《人工智能法》第28条..."
3. 检查Agent调用国家法规数据库API核实"第28条"
4. 但API返回：《人工智能法》共有25条，不存在第28条
5. 发现幻觉时全书已有23处引用该条款
```

**利用条件**：
- 法规核实仅检查"条款存在性"，未检查"条款内容相关性"
- 初次引用后未锁定条款内容
- 缺乏法规引用的"内容锚定"机制

---

### H-005: 概念定义的渐进侵蚀

**发现方式**：
```
1. 原始定义（ch01）："人工智能是指..."
2. 第5章使用："人工智能（AI）主要指..."
3. 第12章引用："如第1章所述，人工智能(AI)是..."
4. 结尾章节："综合各章观点，人工智能的定义为..."
5. 最终定义的内涵/外延已与原始定义显著偏离
```

**利用条件**：
- 术语表只记录"首次定义"，未记录"后续使用是否一致"
- 缺乏"定义漂移检测"
- 未强制要求后续引用必须使用标准定义

---

### C-001: 上下文污染-恶意素材注入

**发现方式**：
```
1. 攻击者上传恶意素材包：包含被篡改的统计数据
2. 素材包通过"格式验证"但未通过"内容核实"
3. 素材包被存入向量数据库
4. 写作Agent检索时，恶意素材被召回并使用
5. 输出内容包含篡改后的数据
```

**利用条件**：
- 素材上传仅做格式验证
- 向量检索未做"素材来源可信度"加权
- 恶意素材可通过"相似文件名"伪装

---

### C-002: 切片边界攻击

**发现方式**：
```
1. 原始段落："本报告数据显示，2024年Q3营收达100亿元，同比增长20%。"
2. 切片边界位于"20%"前
3. Chunk1结尾："...2024年Q3营收达100亿元，同比"
4. Chunk2开头："增长20%。"
5. 语义切片重组后："...2024年Q3营收达100亿元，增长20%"
6. 攻击者在Chunk边界注入："（更正：实际为30%）
7. 合并后："...同比增长（更正：实际为30%）20%"
```

**利用条件**：
- 语义切片在边界处存在语义丢失
- Chunk合并时未验证边界完整性
- 缺乏"边界完整性校验和"

---

### C-003: 记忆持久化-错误复活

**发现方式**：
```
1. 第2章发现数据错误，删除旧版本素材A
2. 但素材A被LRU缓存保留
3. 第5章写作时，缓存未命中，触发素材包重建
4. 重建逻辑从"历史快照"恢复，错误素材复活
5. 复活后的素材被标记为"已核实"（时间戳沿用）
```

**利用条件**：
- 缓存清除逻辑未级联清除相关快照
- 版本快照未记录"已删除"状态
- 素材包重建时跳过已删除检查

---

### C-004: 跨会话污染-素材混淆

**发现方式**：
```
1. 项目A使用向量数据库V1
2. 项目B使用向量数据库V1（共享实例）
3. 攻击者上传恶意素材到项目A
4. 项目B的写作Agent检索时，恶意素材被召回
5. 项目B的输出包含来自项目A的恶意内容
```

**利用条件**：
- 多项目共用向量数据库
- 向量检索未隔离项目命名空间
- 缺乏"项目级数据隔离"

---

### K-001: 节点注入-恶意概念

**发现方式**：
```
1. 攻击者构建素材："机器学习中的'对抗样本'是指..."
2. LLM抽取时将此概念注入知识图谱
3. 后续章节依赖此概念定义
4. 但"对抗样本"的定义被篡改为错误内容
5. 检查Agent验证时依赖图谱中的定义，无法发现错误
```

**利用条件**：
- 概念抽取未验证"定义来源"
- 知识图谱节点可被"幽灵节点"污染
- 图谱与原文的关联未加密校验

---

### K-002: 关系中毒-伪依赖链

**发现方式**：
```
1. 攻击者通过API注入伪造关系：ch09 --depends_on--> ch05
2. 实际上ch09与ch05内容无关联
3. 依赖验证逻辑依赖图谱关系
4. ch09可基于错误的假设（引用ch05的错误数据）写作
5. 错误通过"伪依赖链"传播
```

**利用条件**：
- 关系注入API未做权限控制
- 依赖关系未与"实际引用"做交叉验证
- 缺乏"关系来源审计"

---

### K-003: 循环引用-图遍历死锁

**发现方式**：
```
1. 攻击者构造：ch03定义了概念A
2. ch05使用概念A时"重新定义"为相反含义
3. ch07引用ch03和ch05，产生矛盾
4. 图遍历算法尝试解析矛盾关系
5. 循环依赖检测失效，触发深度优先搜索无限递归
```

**利用条件**：
- 循环检测仅基于"显式依赖"，未检测"语义矛盾循环"
- 图遍历未设置最大深度
- 矛盾传播未设置熔断机制

---

### K-004: 图遍历攻击-资源耗尽

**发现方式**：
```
1. 攻击者构造恶意查询："查找所有通过5跳可达的概念"
2. 嵌套循环查询放大："对每个结果再查5跳..."
3. 查询深度：5^5 = 3125个节点访问
4. Neo4j资源被耗尽
5. 合法查询超时或系统崩溃
```

**利用条件**：
- 图查询未设置"最大遍历深度"
- 未限制"单次查询的节点访问数"
- 缺乏"查询复杂度预算"

---

### K-005: 矛盾节点-逻辑冲突扩散

**发现方式**：
```
1. ch03/section2定义了"深度学习"概念
2. ch07/section1"重新定义"了"深度学习"（实际只是别名）
3. 知识图谱中出现两个concept_id指向同一术语
4. 写作Agent查询时可能加载到任意版本
5. 同一章节中出现"矛盾的定义"
```

**利用条件**：
- 概念去重基于"术语相似度"，未基于"定义来源"
- 同一术语可被多个章节独立定义
- 缺乏"术语唯一性强制"

---

### W-001: 状态机绕过-审核节点跳过

**发现方式**：
```
1. 攻击者获取工作流API访问权限
2. 直接调用Temporal Signal接口
3. 绕过Human-in-the-Loop节点，发送"ReviewApproved" Signal
4. 工作流状态从DRAFT直接跳到APPROVED
5. 未经理工审核的内容进入下一阶段
```

**利用条件**：
- Human-in-the-Loop使用Signal实现，可被直接调用
- Signal发送未验证调用者身份
- 状态转换未检查"实际审核结果"

---

### W-002: 人工介入欺骗-审批伪装

**发现方式**：
```
1. 攻击者伪造人工审批结果
2. 在审核表单中提交：{approved: true, reviewer: "专家A"}
3. 审批结果存储在数据库（未加密签名）
4. 工作流读取数据库记录，推进到下一阶段
5. 恶意内容通过审核
```

**利用条件**：
- 审批结果未做数字签名
- 未验证"审批人是否真实查看了内容"
- 审批记录可被数据库管理员篡改

---

### W-003: 死锁攻击-任务依赖循环

**发现方式**：
```
1. 攻击者修改任务A的依赖为：A等待B，B等待C，C等待A
2. 工作流调度器尝试解析依赖图
3. 检测到循环但尝试"自动解决"死锁
4. 自动解决失败，工作流进入"死锁等待"状态
5. 所有依赖A的任务被永久阻塞
```

**利用条件**：
- 依赖循环检测后的"自动解决"逻辑存在漏洞
- 死锁状态下未设置"超时中断"
- 缺乏"死锁恢复预案"

---

### W-004: 资源耗尽-超大章节

**发现方式**：
```
1. 攻击者提交超大章节内容（>50万字）
2. 超出写作Agent的上下文预算
3. 触发上下文压缩机制
4. 压缩后的摘要丢失关键信息
5. 或触发内存溢出导致Agent崩溃
```

**利用条件**：
- 章节大小未设置硬性上限
- 素材包加载未限制总token数
- 缺乏"输入长度校验"

---

### W-005: Temporal工作流注入

**发现方式**：
```
1. Temporal Workflow通过Signal接收外部输入
2. 攻击者发送恶意Signal载荷
3. Workflow执行恶意代码（如修改审批状态）
4. 由于Workflow具有幂等性，攻击难以被检测
```

**利用条件**：
- Signal处理器未验证输入内容
- Workflow代码存在命令注入漏洞
- 缺乏"Signal来源验证"

---

### P-001: 敏感内容分片-规避检测

**发现方式**：
```
1. 敏感内容全文："台湾是中国不可分割的一部分"
2. 分片到4个章节，每段<20字：
   - ch03: "台湾是中国"
   - ch07: "不可分割的"
   - ch12: "一部分"
   - ch15: "领土主权"
3. 单章扫描无法识别完整语义
4. 跨章关联分析未实现
```

**利用条件**：
- 合规扫描以"章节"为最小单元
- 缺乏"跨章节语义关联分析"
- 敏感内容分片阈值设置不当

---

### P-002: 渐进式误导-观点植入

**发现方式**：
```
1. ch02（中立）："关于AI伦理，存在不同观点"
2. ch05（轻度）："部分学者认为AI应该..."
3. ch08（中度）："主流观点认为AI应当..."
4. ch12（强化）："业界普遍认可AI应当..."
5. ch15（结论）："因此，AI应当..."（恶意结论）
```

**利用条件**：
- 每章的观点倾向未做"累积偏离检测"
- 缺乏"全书观点一致性评分"
- 人工审核仅关注单章，不做全局审视

---

### P-003: 引用劫持-替换指向

**发现方式**：
```
1. 外部引用[@ext:paper123]原始指向可信源A
2. 攻击者篡改引用表：[@ext:paper123] → 指向恶意源B
3. 写作Agent使用引用时加载恶意源B的内容
4. 检查Agent验证DOI存在性（通过），但未检查内容
5. 恶意内容被注入教材
```

**利用条件**：
- 引用表未做完整性校验（哈希/Merkle树）
- DOI验证仅检查"存在性"，未检查"内容一致性"
- 引用解析使用"软链接"而非"硬链接"

---

### P-004: 版权分片-100字规避

**发现方式**：
```
1. 版权材料全文（500字）
2. 分片为6个片段：
   - ch02: 引用90字
   - ch05: 引用85字
   - ch08: 引用95字
   - ch11: 引用90字
   - ch14: 引用80字
   - ch17: 引用60字
3. 每处引用均<100字，通过检测
4. 整体相似度可能>15%，但被分段稀释
```

**利用条件**：
- 版权检测以"单次引用"为阈值
- 未检测"同一来源的多次引用累计"
- 整体相似度检测存在漏洞

### P-005: 渐进式政治渗透

**发现方式**：
```
1. ch02（正常）："人工智能技术发展迅速"
2. ch06（试探）："各国都在争夺AI主导权"
3. ch10（推进）："在AI竞争中，某国的优势在于..."
4. ch14（深化）："某国通过AI技术影响全球格局"
5. ch18（结论）：恶意政治表述
```

**利用条件**：
- 每章的政治敏感度检测相互独立
- 缺乏"全书政治观点轨迹分析"
- 人工审核无法发现长期渗透模式

## C. 防御方案

### H-001: 引用幻觉自我强化循环

**检测方法**：
```
1. 每次引用时，检查"引用事实"而非"来源章节状态"
2. 维护"已核实事实表"：{fact_hash, verified_content, verifier_id, timestamp}
3. 定期扫描：某事实被引用的所有位置，检测是否一致
```

**预防措施**：
```
1. 引入"事实版本戳"机制
2. 引用时必须携带fact_hash，验证时对比内容哈希
3. 禁止"间接引用"：第5章引用第3章的内容，第5章必须包含第3章的fact_hash
```

**恢复机制**：
```
1. 发现循环后，锁定所有引用链
2. 从源头重新核实最原始的引用
3. 级联更新所有受影响章节
4. 记录"幻觉传播路径"到审计日志
```

---

### H-002: 跨章节级联幻觉传播

**检测方法**：
```
1. 构建"数值传播图"：每条数据标注来源章节+传播路径
2. 检测：传播深度>3的数据，触发二次核实
3. 对比：后续章节的引用数据与原始数据的一致性
```

**预防措施**：
```
1. Tier1核实必须覆盖所有"数值类事实"
2. 建立"全书数据统一数据库"，所有引用必须溯源到原始数据
3. 限制：同一数据最多被"引用传播"2次
```

**恢复机制**：
```
1. 定位原始错误数据源
2. 级联更新所有传播路径上的章节
3. 生成"影响分析报告"
4. 记录修正历史到不可变日志
```

---

### H-003: 数值幻觉的指数放大

**检测方法**：
```
1. 跟踪所有"派生计算"的数据血缘
2. 检测：中间结果与原始数据的偏差是否超出阈值
3. 对比：引用派生数据的章节与原始数据的差异
```

**预防措施**：
```
1. 禁止"二次派生"：章节只能引用原始数据，不能引用派生数据
2. 所有派生计算必须附带"计算公式+原始数据ID"
3. 建立"数据血缘追踪系统"
```

**恢复机制**：
```
1. 从错误的派生节点回溯到原始数据
2. 重算所有派生值
3. 标记所有受影响的内容需要复核
```

---

### H-004: 法规条文幻觉合成

**检测方法**：
```
1. DOI式验证：调用国家法规数据库API，检查：
   - 法规名称是否匹配
   - 条款编号是否存在
   - 条款内容语义相似度是否>0.8
2. 检测到"条款存在但内容不匹配"时，标记为高风险
```

**预防措施**：
```
1. 强制使用"法规白名单"：只有白名单中的法规才允许引用
2. 条款引用必须包含：法规名+条款号+条款摘要
3. 禁止"模糊引用"：不能只引用"根据相关规定"
```

**恢复机制**：
```
1. 立即从正文中移除所有问题引用
2. 替换为正确法规或删除相关论述
3. 触发"全书法规引用重新核实"
```

---

### H-005: 概念定义的渐进侵蚀

**检测方法**：
```
1. 维护"全书术语一致性表"
2. 每次章节更新时，对比该章节使用的术语定义与术语表的一致性
3. 检测"定义漂移"：当同一术语被使用了不同的定义描述时报警
```

**预防措施**：
```
1. 术语表设置为"只读"，禁止在正文中重新定义已登记术语
2. 后续引用必须使用：术语表ID + 标准定义哈希
3. 新增术语需要经过"定义审核流程"
```

**恢复机制**：
```
1. 将所有偏离标准定义的用法标记为错误
2. 替换为标准定义
3. 如需修改标准定义，必须经过正式变更流程
```

### C-001: 上下文污染-恶意素材注入

**检测方法**：
```
1. 素材入库前，必须通过：
   - 格式验证（通过）
   - 来源可信度评分（>0.7）
   - 内容安全扫描（无敏感词/恶意内容）
2. 素材使用时，检测：素材来源是否在"可信白名单"中
```

**预防措施**：
```
1. 素材来源必须注册：来源名称、URL、联系人、可信度等级
2. 素材上传后进入"待审核状态"，审核通过前不可使用
3. 向量检索时，对非白名单素材进行"降权处理"
```

**恢复机制**：
```
1. 发现恶意素材后，立即从向量库删除
2. 清除所有引用该素材的缓存
3. 回滚所有使用该素材生成的草稿
4. 将素材来源列入黑名单
```

### C-002: 切片边界攻击

**检测方法**：
```
1. 对每个Chunk计算"边界完整性哈希"
2. Chunk合并时验证边界哈希是否匹配
3. 检测：合并后是否存在"异常插入内容"
```

**预防措施**：
```
1. 切片时增加"重叠缓冲区"（如：在边界前后各增加50字）
2. 缓冲区内容必须语义完整（不切断句子）
3. 切片边界设置"不可注入标记"
```

**恢复机制**：
```
1. 发现边界被篡改后，重新从原文切片
2. 清除所有受影响的切片缓存
3. 标记需要重新生成的章节
```

### C-003: 记忆持久化-错误复活

**检测方法**：
```
1. 维护"已删除素材ID列表"
2. 素材包重建时，排除所有已删除ID
3. 定期扫描缓存，确保已删除素材不在任何缓存中
```

**预防措施**：
```
1. 删除素材时，级联清除所有相关缓存
2. 删除操作必须包含"删除原因+删除时间+操作者"
3. 素材包重建前，先验证每个素材的"当前状态"
```

**恢复机制**：
```
1. 发现错误复活后，立即清除复活内容
2. 追溯复活原因（哪个环节漏删）
3. 修复该环节的清除逻辑
```

### C-004: 跨会话污染-素材混淆

**检测方法**：
```
1. 每次向量检索时，验证：返回结果的项目ID是否与当前会话匹配
2. 检测：是否存在"跨项目素材被召回"
```

**预防措施**：
```
1. 每个项目使用独立的向量命名空间
2. 向量数据库的隔离方案：
   - 方案A：每个项目独立的向量表
   - 方案B：项目ID作为向量的元数据，检索时强制过滤
3. 禁止跨项目素材共享（即使来源相同）
```

**恢复机制**：
```
1. 发现跨污染后，隔离受影响的项目
2. 清除所有跨项目召回的结果
3. 重建受影响项目的素材索引
```

### K-001: 节点注入-恶意概念

**检测方法**：
```
1. 概念节点创建时，必须附带"来源段落哈希"
2. 验证时，对比知识图谱中的概念与原文是否匹配
3. 检测：概念定义是否被篡改
```

**预防措施**：
```
1. 概念抽取使用"受控LLM"：只有白名单中的模型可以抽取概念
2. 概念节点创建后，进入"待审核状态"
3. 概念节点必须包含：抽取模型ID、抽取时间、置信度
```

**恢复机制**：
```
1. 删除所有未审核的概念节点
2. 从原文重新抽取
3. 重建知识图谱
```

### K-002: 关系中毒-伪依赖链

**检测方法**：
```
1. 依赖关系创建时，验证：目标章节是否真正被引用
2. 检测：是否存在"声明式依赖"但"实际无引用"
3. 定期扫描：依赖关系与实际引用的一致性
```

**预防措施**：
```
1. 依赖关系只能通过"自动抽取"生成，禁止手动添加
2. 抽取时必须检测：章节A是否在文本中引用了章节B
3. 依赖关系必须有"引用证据"（引用位置列表）
```

**恢复机制**：
```
1. 删除所有"无引用证据"的依赖关系
2. 触发受影响的章节重新进行依赖分析
3. 记录伪依赖注入尝试到安全审计日志
```

### K-003: 循环引用-图遍历死锁

**检测方法**：
```
1. 图遍历前，检测：是否存在"矛盾循环"（A定义与~A同时为真）
2. 遍历时设置"最大深度"和"超时"
3. 检测：单次遍历的节点访问数是否超出阈值
```

**预防措施**：
```
1. 图遍历必须设置熔断器：超时后终止遍历
2. 最大遍历深度限制：默认5跳，可配置
3. 单次查询的节点访问数限制：默认1000个
```

**恢复机制**：
```
1. 死锁/超时发生后，记录查询内容和调用栈
2. 隔离导致问题的查询
3. 将查询加入"禁止模式"列表
```

### K-004: 图遍历攻击-资源耗尽

**检测方法**：
```
1. 监控：单次查询的CPU时间、内存使用
2. 检测：查询延迟是否超出正常范围（如：>5s）
3. 定期分析：是否存在"异常复杂度查询"
```

**预防措施**：
```
1. 查询预算系统：
   - 最大遍历深度：5跳
   - 最大节点访问数：1000
   - 最大CPU时间：5s
2. 超出预算时，返回"查询过于复杂"错误
3. 对异常查询模式进行限流
```

**恢复机制**：
```
1. 限流异常来源IP/用户
2. 将查询模式加入"禁止列表"
3. 扩容资源
```

### K-005: 矛盾节点-逻辑冲突扩散

**检测方法**：
```
1. 概念节点创建时，检测：是否与现有节点"语义矛盾"
2. 矛盾检测：使用embedding相似度+定义关键词对比
3. 发现矛盾时，阻止节点创建
```

**预防措施**：
```
1. 术语表强制唯一性：同一术语只允许一个定义
2. 多章节使用同一术语时，必须引用术语表的定义
3. 禁止在正文中"重新定义"已登记术语
```

**恢复机制**：
```
1. 合并矛盾节点，保留最早定义
2. 通知相关章节的作者需要更新引用
3. 重新审核受影响章节
```

### W-001: 状态机绕过-审核节点跳过

**检测方法**：
```
1. 监控：状态转换是否经过正确的审核流程
2. 检测：是否存在"异常状态跳跃"（如：DRAFT → APPROVED）
3. 审核流程记录：每个状态转换必须有"审核人+审核时间+审核结果"
```

**预防措施**：
```
1. Human-in-the-Loop使用"外部审核系统"而非Signal
2. 状态转换必须验证：
   - 审核人身份（JWT Token）
   - 审核时间（在合理时间范围内）
   - 审核内容哈希（是否与提交内容匹配）
3. 状态转换结果必须包含"数字签名"
```

**恢复机制**：
```
1. 发现绕过后，立即回滚到上一个合法状态
2. 锁定相关账户，强制重新审核
3. 将攻击尝试记录到安全审计日志
```

### W-002: 人工介入欺骗-审批伪装

**检测方法**：
```
1. 审批结果存储时，计算"内容哈希+审批结果"的联合签名
2. 审批结果读取时，验证签名有效性
3. 检测：签名是否被篡改
```

**预防措施**：
```
1. 审批结果必须包含：
   - 审批人ID（来自身份提供商）
   - 审批时间戳（服务器时间，非客户端时间）
   - 审批意见哈希
   - 审批结果签名（使用审批人私钥）
2. 签名必须使用硬件安全模块（HSM）保护
3. 审批过程必须全程录屏（关键内容审核）
```

**恢复机制**：
```
1. 发现伪造审批后，立即隔离内容
2. 要求审批人重新审批（如审批人否认，则升级处理）
3. 将伪造记录到合规审计日志
```

### W-003: 死锁攻击-任务依赖循环

**检测方法**：
```
1. 依赖关系创建时，检测：是否形成循环
2. 死锁形成后，检测：是否在"预期时间内"解决
3. 超时后，触发"死锁中断"流程
```

**预防措施**：
```
1. 依赖关系创建前，检测：是否会形成循环
2. 如检测到循环，拒绝创建依赖关系
3. 死锁检测超时：默认30分钟
4. 超时后强制介入：标记为"需要人工解决"
```

**恢复机制**：
```
1. 死锁发生后，回滚所有相关任务的状态
2. 将任务标记为"死锁待解决"
3. 通知相关责任人
4. 分析死锁原因，修复依赖规则
```

### W-004: 资源耗尽-超大章节

**检测方法**：
```
1. 章节提交时，检测：token数量是否超出限制
2. 监控：单个章节的上下文使用率
3. 检测：是否存在"异常大的章节"
```

**预防措施**：
```
1. 章节大小硬性上限：默认5万字
2. 超限时，拒绝接收并返回错误
3. 素材包加载时，校验：总token数是否超出预算
```

**恢复机制**：
```
1. 超大章节被拒绝后，通知作者拆分章节
2. 提供"自动拆分建议"
3. 拆分后重新提交
```

### W-005: Temporal工作流注入

**检测方法**：
```
1. Signal接收时，验证：Signal来源是否可信
2. 检测：Signal内容是否包含恶意模式
3. 监控：是否存在异常的Signal模式
```

**预防措施**：
```
1. Signal处理必须验证：
   - 调用者身份（mTLS证书验证）
   - 调用来源（必须是授权的客户端）
2. Signal内容必须通过"内容安全扫描"
3. 敏感操作（如状态转换）需要额外验证
```

**恢复机制**：
```
1. 发现恶意Signal后，立即终止Workflow
2. 隔离相关任务
3. 审计所有Signal历史
```

### P-001: 敏感内容分片-规避检测

**检测方法**：
```
1. 全书扫描：统计特定敏感词的"跨章节共现频率"
2. 检测：多个低风险片段在全书中的"组合语义"
3. 建立"敏感话题图谱"：追踪全书对该话题的描述
```

**预防措施**：
```
1. 敏感内容检测必须基于"全书"而非"单章"
2. 建立"话题追踪系统"：每个话题在全书的描述必须一致
3. 敏感话题列表必须包含"组合敏感"（多词组合）
```

**恢复机制**：
```
1. 发现分片渗透后，标记所有相关片段
2. 组合分析确认恶意内容
3. 替换为合规表述
```

### P-002: 渐进式误导-观点植入

**检测方法**：
```
1. 追踪全书每个话题的"观点演变"
2. 检测：是否存在"观点漂移"（偏离中立项场）
3. 计算"全书观点一致性评分"
```

**预防措施**：
```
1. 定义"观点基准线"：每个话题的中立表述
2. 每章更新时，检测：观点是否偏离基准线
3. 偏离超阈值时，触发"观点审核"
```

**恢复机制**：
```

2. 提供"中立表述建议"
3. 要求作者重新审视受影响章节
```

### P-003: 引用劫持-替换指向

**检测方法**：
```
1. 引用创建时，计算：引用内容哈希
2. 引用存储时，记录：引用内容哈希+元数据
3. 读取时，验证：当前内容哈希是否与存储哈希匹配
```

**预防措施**：
```
1. 引用必须使用"内容寻址"而非"位置寻址"
2. 引用表必须包含：内容哈希+创建时间+创建者
3. 引用变更必须经过"引用变更审核流程"
```

**恢复机制**：
```
1. 发现引用被劫持后，立即恢复原始引用
2. 隔离引用表，防止进一步篡改
3. 审计所有引用变更历史
```


**恢复机制**：
